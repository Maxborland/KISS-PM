# Worker 04: project calendars write E2E

## Scope

Only `e2e/full-eval/projects-calendars-write.spec.ts` is adapted. Product code is unchanged.

## Behavior spec

- AC1: ADMIN creates a project-level non-working calendar exception through the calendar grid UI; the UI first POSTs to the exact `preview-command` endpoint, shows `Предпросмотр изменений`, and only POSTs to `apply-command` after explicit confirmation.
- AC2: The preview and apply envelopes match, the create command succeeds, the planning read-model contains the exact exception, and the exception remains visible after reload.
- AC3: ADMIN removes the exception through the exception-list UI control using the same required `preview-command` -> preview dialog -> `apply-command` sequence.
- AC4: The preview and apply envelopes match, the removal command succeeds, the read-model marks the record inactive by restoring full working minutes, and the exception remains absent after reload.
- AC5: A `finally` cleanup restores full working minutes through the public API if the UI flow fails after creation.
- AC6: `plan-reader-no-resources` can read the calendar but receives `403` from the exact `preview-command` endpoint; no preview dialog or `apply-command` request follows, and plan version and exceptions remain unchanged after reload.

## Test mapping

- AC1-AC5 -> `ADMIN creates and removes a project calendar exception with API readback and reload`
- AC6 -> `PLAN reader cannot create a project calendar exception`

## Selector and data strategy

- The test selects the first enabled button with the existing title `Рабочий день — клик: нерабочий`, avoiding seeded dates that are already weekends or active exceptions.
- The exact exception `id`, `date`, and `calendarId` are read from the POST generated by the real UI, then used for API and reload assertions.
- Planning response helpers compare the URL pathname to an exact endpoint path; substring matching cannot confuse `preview-command` with `preview-command-batch`.
- The read-only test waits for the grid control to become enabled again after the denied preview before asserting that neither exact apply endpoint (`apply-command` nor `apply-command-batch`) was requested.
- Removal is modeled according to the product contract: `calendar.exception.upsert` restores `workingMinutesPerDay`; it does not physically delete the persisted row.
