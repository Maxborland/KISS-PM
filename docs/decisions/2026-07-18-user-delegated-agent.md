# ADR: Агент действует строго от имени пользователя

Дата: 2026-07-18.

## Статус

Принято.

## Контекст

Агент «Генри Гантт» умеет читать данные (analyze) и предлагать изменения (mutation):
задачи, комментарии, применение planning-сценариев и команд плана. Нужно было решить,
с какими правами он работает: как системный/сервисный аккаунт с собственными широкими
правами или строго как делегат конкретного пользователя. Принципы честности проекта
требуют, чтобы отказ прав был виден в audit trail, а применения происходили только
через preview→apply с явным подтверждением.

## Решение

Агент — user-delegated: никакого сервисного аккаунта, все действия — сессия и RBAC
пользователя (`apps/api/src/agent/agentRoutes.ts`, `toolRegistry.ts`):

- Каждый agent-роут (`/tools`, `/propose`, `/propose/stream`, `/execute`, `/thread`)
  начинается с `getSessionActorFromHeaders(cookie)`; без сессии — 401. Отдельного
  системного актора в кодовой базе агента нет.
- Набор инструментов фильтруется по правам актора (`allowedToolsForActor`: каждый tool
  несёт `capability` из `@kiss-pm/access-control` — `canReadProjects`,
  `canApplyPlanningScenarios` и т. д.). LLM получает только разрешённые инструменты;
  системный промпт фиксирует: «ты не можешь делать то, что сотруднику не разрешено».
- Исполнение — внутренняя переотправка в те же governed-эндпоинты, что зовёт человеческий
  UI (`dispatchInternal`/`dispatchBinding` с cookie пользователя): RBAC, транзакция,
  version-lock, integrity и audit переиспользуются как есть, бизнес-логика не дублируется.
  Вложения читаются штатным download-роутом под той же cookie — недоступное по RBAC
  вложение честно маркируется, а не пропускается молча.
- Propose и execute разделены: в LLM-цикле mutation-инструменты никогда не исполняются —
  только предлагаются. Offerable — лишь те mutation (`OFFERABLE_MUTATIONS`), для которых
  review-карточка показывает полный честный before/after (payload-backed);
  `apply_plan_commands` исполняем вручную, но не offerable, пока его generic-preview
  недостаточно честен. Применение — отдельный `POST /execute` по подтверждённым
  пользователем действиям: грубая `capability`-проверка (defense-in-depth) плюс точная
  проверка внутри governed-команды.
- Preconditions передаёт клиент, fail-closed: `taskUpdatedAt` / `planVersion` обязаны
  прийти в `preconditionVersions` из review-карточки (или явно в input); отсутствие —
  400 `missing_precondition_versions`. Серверная подстановка текущей версии запрещена —
  она обесценила бы optimistic lock.
- Провенанс: на каждое применённое или отклонённое правами действие пишется отдельное
  audit-событие `agent.<tool>.applied|denied` с `sourceWorkflow: "agent"` и общим
  `correlationId` батча — попытки агента, отклонённые RBAC, не исчезают из governance
  trail. Квитанция fail-closed: id событий попадают в ответ только если audit-запись
  реально произошла.
- Стоимость под контролем той же личности: лимит 2 одновременных LLM-циклов на
  `(tenant:user)` против denial-of-wallet.

## Рассмотренные альтернативы

- **Сервисный аккаунт / системный актор с расширенными правами.** Агент мог бы делать
  больше, чем пользователь, — это ломает модель ответственности (в audit значился бы не
  тот, кто решил), требует параллельной матрицы прав и открывает privilege escalation
  через формулировку запроса к LLM.
- **Автоисполнение mutation прямо в LLM-цикле.** Быстрее, но нарушает preview→apply:
  пользователь одобрял бы результат постфактум, а не действие заранее; отвергнуто.
- **Серверная подстановка актуальных версий при execute.** Удобнее для клиента, но
  превращает optimistic concurrency в формальность: пользователь подтверждал карточку
  против одного состояния, а применилось бы к другому.

## Последствия

- Агент по построению не может превысить права пользователя: и фильтрация инструментов,
  и повторная проверка на execute, и точный RBAC внутри governed-роутов работают от
  одной и той же сессии.
- Отказ прав — наблюдаемое событие (`agent.*.denied` в audit), а не молчаливое отсутствие
  результата; per-item outcomes (`applied/denied/conflict/failed`) видны в квитанции.
- Двойная RBAC-проверка (грубая capability + точная в команде) — осознанная избыточность
  (defense-in-depth), её нужно поддерживать при добавлении инструментов.
- Конфликт версий — штатный исход (`conflict` в outcomes, `currentVersions` в ответе):
  клиент обязан перепоказать карточку, а не ретраить молча.
- Делегирование «на будущее» (действия без активной сессии пользователя, по расписанию)
  в эту модель не входит и потребует отдельного ADR.

## Acceptance evidence

- `/execute` без `preconditionVersions` для `change_task_status` / `update_task` /
  planning-применений возвращает 400 `missing_precondition_versions`.
- Denied-действие даёт audit-событие `agent.<tool>.denied` с
  `permissionResult.allowed = false` и `via: "agent"`.
- Планообразующие применения возвращают в квитанции `planningAuditEventId` /
  `planVersion` из governed-ответа — ссылки только на реальные записи.
