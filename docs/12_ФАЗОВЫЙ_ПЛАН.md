# 12. Фазовый план реализации

## Принцип

Новая реализация начинается только после утверждения этого docs baseline. Не строим хаотичный MVP. Каждая фаза имеет закрытый backlog, non-scope, acceptance criteria и E2E.

## Phase 0. Документальный baseline

Цель: утвердить русскоязычные source-of-truth документы, reference alignment и границы первой реализации.

Exit gate: документы согласованы, открытые решения перечислены, старый код удален из рабочей копии.

## Phase 1. Платформа и каркас

Цель: создать новый monorepo, app shell, API shell, PostgreSQL, migrations, auth/session spike, базовый E2E smoke.

Non-scope: Gantt, KPI, resource matrix.

## Phase 2. Tenant, users, access, config

Цель: tenant isolation, пользователи, роли, access profiles, custom fields, templates basics.

Exit gate: tenant A не видит tenant B; admin меняет label/custom field без кода.

Практическая последовательность после Phase 2.1: сначала single-workspace auth/RBAC/user foundation, затем возврат к tenant separation и SaaS/operator admin. Это защищает продукт от прежней ошибки, когда строилась админка платформы раньше рабочего пользовательского контура.

## Phase 3. CRM intake -> active project

Цель: ручная opportunity, readiness, demand `должность + часы`, capacity check по доступным часам должностей и активация проекта.

Exit gate: пользователь создает opportunity, видит blockers/feasibility, активирует проект через governed action и получает audit.

### Phase 3.1. CRM foundation and deal UX cleanup

Цель: до перехода к проектному контуру сделать сделки полноценным входом в проект: клиент, контакт, тип проекта, этап сделки, детали сделки, list/kanban views и аккуратный рабочий UI без фейковых действий.

Exit gate: пользователь создает клиента, контакт, тип проекта и сделку; открывает детали сделки; видит сделку в списке и kanban; меняет этап реальным action; Phase 3 feasibility/activation остается рабочей; RBAC/audit/tenant isolation доказаны тестами и smoke.

## Phase 4. Project lifecycle, tasks, My Work

Цель: active project, stages, canonical task, participant roles, My Work/Kanban.

Exit gate: одна задача видна в project, My Work и control context; роль пользователя влияет на view/actions.

Starter-срез Phase 4 зафиксирован в `24_PHASE_4_PROJECT_LIFECYCLE_TASKS_MY_WORK.md`: сначала добавляем tenant-scoped `Task`, участников, детали активного проекта, создание задачи, `Моя работа` и audit, без Gantt/WBS/dependencies/baseline.

### Phase 4.2. Storage and connector foundation

Цель: после общего `CrmActivity` и starter-задач добавить единый слой файловых объектов, внешних ссылок и connector references, который сможет обслуживать CRM-карточки, задачи проекта, будущие документы, импорты и control surfaces.

Exit gate: пользователь прикрепляет файл или внешнюю ссылку к CRM-сущности или задаче через permissioned action; система хранит tenant-scoped `FileAsset`/`ExternalReference`, пишет audit, проверяет безопасные URL/metadata и не смешивает binary storage с CRM activity.

Этот slice фиксируется отдельно в `26_PHASE_4_2_STORAGE_CONNECTOR_FOUNDATION.md` и идет до тяжелого Gantt/WBS, потому что вложения и внешние источники станут общим инфраструктурным контрактом для проектного контура.

## Phase 5. Gantt/WBS scheduling foundation

Цель: WBS, Gantt grid/timeline, dependencies MVP, baseline basics, task edits through commands.

Exit gate: Gantt action проходит права, audit и reload.

Архитектурная закладка для будущего scenario planning: scheduling engine должен принимать snapshot плана и возвращать детерминированный расчет дат, validation issues и read model, чтобы Phase 6 могла использовать его для preview сценариев. Сам optimizer в Phase 5 не входит.

## Phase 6. Resource matrix and conflict loop

Цель: resource load matrix, availability exceptions, free capacity, overload drilldown, preview resolution.

Exit gate: E2E показывает перегруз -> drilldown -> три scenario proposals -> выбор сценария -> action -> audit -> recalculation.

Scenario planning MVP: система предлагает aggressive, balanced и resilient варианты решения перегруза на одном проекте. Разрешенные операции: shift, split, reassign, reserve, accept risk и move deadline. Применение только через governed action.

## Phase 7. KPI, signals, action engine

Цель: KPI definitions, evaluations, control signals, corrective actions, action engine, audit viewer.

Exit gate: KPI deviation превращается в governed action и audit.

Scenario planning после Phase 6 должен стать источником управленческих предложений для schedule/resource control signals: сигнал может вести пользователя не только к ручному действию, но и к сравнению сценариев с влиянием на KPI, дедлайны и approvals.

## Phase 8. Control surfaces builder

Цель: configurable management instruments, saved views, fields, filters, widgets, actions, preview/publish.

Exit gate: tenant admin публикует surface; обычный user работает с ней по правам.

## Phase 9. Closure and retrospectives

Цель: closure workflow, immutable snapshots, plan/fact, lessons, template improvement action.

Exit gate: closed project snapshot влияет на будущую оценку через governed improvement.

## Phase 10. Production hardening

Цель: security, privacy, performance, backup, deployment, self-hosted packaging, monitoring, docs for operators.

Exit gate: полный E2E контур проходит в release-like окружении.
