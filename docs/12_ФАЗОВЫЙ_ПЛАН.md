# 12. Фазовый план реализации

## Принцип

Новая реализация начинается только после утверждения этого docs baseline. Не строим хаотичный MVP. Каждая фаза имеет закрытый backlog, non-scope, acceptance criteria и E2E.

## Phase 0. Документальный baseline

Цель: утвердить русскоязычные source-of-truth документы, reference alignment и границы первой реализации.

Exit gate: документы согласованы, открытые решения перечислены, старый код удален из рабочей копии.

## Phase 1. Платформа и каркас

Цель: создать новый monorepo, app shell, API shell, PostgreSQL, migrations, auth/session spike, базовый E2E smoke.

Non-scope: Gantt, KPI, resource matrix.

## Phase 2. Tenant, users, access, config

Цель: tenant isolation, пользователи, роли, access profiles, custom fields, templates basics.

Exit gate: tenant A не видит tenant B; admin меняет label/custom field без кода.

Практическая последовательность после Phase 2.1: сначала single-workspace auth/RBAC/user foundation, затем возврат к tenant separation и SaaS/operator admin. Это защищает продукт от прежней ошибки, когда строилась админка платформы раньше рабочего пользовательского контура.

## Phase 3. CRM intake -> active project

Цель: ручная opportunity, readiness, demand `должность + часы`, capacity check по доступным часам должностей и активация проекта.

Exit gate: пользователь создает opportunity, видит blockers/feasibility, активирует проект через governed action и получает audit.

### Phase 3.1. CRM foundation and deal UX cleanup

Цель: до перехода к проектному контуру сделать сделки полноценным входом в проект: клиент, контакт, тип проекта, этап сделки, детали сделки, list/kanban views и аккуратный рабочий UI без фейковых действий.

Exit gate: пользователь создает клиента, контакт, тип проекта и сделку; открывает детали сделки; видит сделку в списке и kanban; меняет этап реальным action; Phase 3 feasibility/activation остается рабочей; RBAC/audit/tenant isolation доказаны тестами и smoke.

## Phase 4. Project lifecycle, tasks, My Work

Цель: active project, stages, canonical task, participant roles, My Work/Kanban.

Exit gate: одна задача видна в project, My Work и control context; роль пользователя влияет на view/actions.

Starter-срез Phase 4 зафиксирован в `24_PHASE_4_PROJECT_LIFECYCLE_TASKS_MY_WORK.md`: сначала добавляем tenant-scoped `Task`, участников, детали активного проекта, создание задачи, `Моя работа` и audit, без Gantt/WBS/dependencies/baseline.

### Phase 4.2. Task workspace, статусы задач и карточка задачи

Цель: расширить starter-задачи до полноценного рабочего контура задач: `Моя работа` в режимах список/канбан, CRUD задач, bulk, tenant-настраиваемые статусы, права редактирования, отдельная карточка задачи и task activity.

Exit gate: пользователь создает задачу через модалку, фильтрует и ищет ее в `Моя работа`, переключает список/канбан, открывает карточку задачи, добавляет комментарий, выполняет разрешенный status action, а restricted user не может редактировать задачу. Все write-flow проходят права, audit и reload.

Срез зафиксирован в `28_PHASE_4_2_TASK_WORKSPACE.md`. Редактирование predecessor links остается за future planning UI/API consumer; Phase 5/6 фиксирует только backend planning engine contract и command/read-model слой.

### Phase 4.3. Storage and connector foundation

Статус backend: реализовано как Phase F storage/search backend (PR #23). Документ `26_PHASE_4_2_STORAGE_CONNECTOR_FOUNDATION.md` остается исходным контрактом, а Phase 10 проверяет security/privacy, storage readiness и metadata search leakage.

Цель: после общего `CrmActivity` и starter-задач добавить единый слой файловых объектов, внешних ссылок и connector references, который сможет обслуживать CRM-карточки, задачи проекта, будущие документы, импорты и control surfaces.

Exit gate: пользователь прикрепляет файл или внешнюю ссылку к CRM-сущности или задаче через permissioned action; система хранит tenant-scoped `FileAsset`/`ExternalReference`, пишет audit, проверяет безопасные URL/metadata и не смешивает binary storage с CRM activity.

Этот slice фиксируется отдельно в `26_PHASE_4_2_STORAGE_CONNECTOR_FOUNDATION.md` и идет до тяжелых planning/UI consumers, потому что вложения и внешние источники станут общим инфраструктурным контрактом для проектного контура.

## Phase 5. Planning engine scheduling foundation

Цель: принять backend-решения по deterministic MS Project-class planning engine: canonical `PlanSnapshot`, `PlanningCommand`, task CRUD wrapper, Work/Duration/Units, task types, все типы зависимостей, lag/lead, calendars, constraints, critical path и validation issues.

Exit gate: planning command preview/apply проходит права, plan-version precondition, audit и reload read-model без участия frontend Gantt/resource UI.

Phase 5/6 backend contract закреплен в `30_PHASE_5_6_MS_PROJECT_CLASS_BACKEND.md`: любая `Task` является частью planning model, task CRUD является фасадом над `PlanningCommand`, а scheduling engine сразу проектируется под Work/Duration/Units, task types, все типы зависимостей, lag/lead, calendars, constraints, critical path и validation issues. MSPDI/XML import/export не входит.

## Phase 6. Planning engine resource/scenario layer

Цель: принять backend-решения по resource planning и scenario planning как частям planning engine: load buckets, capacity exceptions, reservations, overload drilldown data, free capacity и explainable `ScenarioProposal` / `PlanDelta`.

Exit gate: backend/API tests доказывают перегруз -> stored scenario proposals -> governed apply -> audit -> recalculation. Gantt UI, resource matrix UI и browser E2E для этих поверхностей не входят в Phase 5/6.

Scenario planning MVP строится поверх MS Project-class scheduling core и resource planning: система предлагает aggressive, balanced и resilient варианты решения перегруза на одном проекте. Разрешенные операции: shift, split, reassign, change units, change duration, reserve, accept risk и move deadline. Применение только через governed action.

## Phase B. Planning Workspace UI (WBS spreadsheet + Gantt)

Цель: MS Project-class spreadsheet parity на TanStack Table + virtualizer, синхронный Gantt, Task Inspector, Preview/Apply bar, server-authoritative `clientPlanVersion` + batch apply, контракт realtime-ready для Phase C.

Non-scope (Phase C): вкладки Ресурсы, Назначения, Календари, Сценарии, Baseline, Аудит проекта, Настройки; SSE/WebSocket push; полный context menu/drag-fill parity E2E.

Exit gate: acceptance из `31_PLANNING_WORKSPACE_UI_DESIGN.md` (раздел spreadsheet) + `docs/32_PHASE_B_PLANNING_UI_DECISIONS.md`; отчет `docs/status/2026-05-23-phase-b-planning-grid.md`.

## Phase C. Planning Workspace доводка (C.1–C.5)

Цель: закрыть вкладки Project Shell (Ресурсы, Назначения, Календари, Сценарии, Baseline, Аудит, Настройки), grid parity (context menu, drag-fill, compensating undo, Excel paste E2E), SSE realtime `planVersionChanged`, narrow fallback и a11y smoke.

Sub-phases: `docs/33` … `docs/37`; closure `docs/status/2026-05-23-phase-c-closure.md`.

Exit gate: C.1 realtime + grid E2E; C.2–C.4 tab routing + panes; C.5 axe + health budgets + `pnpm build`.

## Phase D. Planning Workspace продвинутый контур

Цель: tenant production calendar, месячная resource matrix с cross-project drilldown, `project.settings.update`, saved WBS views, custom task fields, **absences plane** и **tenant org structure** (два трека, матрица 4 уровня, фильтры Users).

План: `docs/38_PHASE_D_PLAN.md`; closure `docs/status/2026-05-23-phase-d-closure.md`. Миграции `0023`–`0026`.

Exit gate: `pnpm test`, `pnpm typecheck`, `pnpm build`, `pnpm test:db`, `pnpm test:e2e:smoke`.

Non-scope: ICS import, approval workflow для absences.

## Phase E. Tenant resource load по оргструктуре

Статус backend: реализовано (PR #22). Phase 10 проверяет tenant isolation, hidden contribution masking, cache invalidation и performance fixture; UI/отдельная поверхность не входит в backend hardening.

Цель: сквозной отчёт/матрица загрузки **по всем проектам** tenant с группировкой по org track (направление → отдел/команда → должность → сотрудник). Внутри одного проекта матрица и overload уже есть в Phase D.

План-детали: `docs/38_PHASE_D_PLAN.md` (секция Phase E). Поверхность TBD: control surface или отдельный workspace route.

Non-scope в Phase E: полный BI-конструктор, экспорт в Excel (если не входит в минимальный slice).

## Phase 7. KPI, signals, action engine

Статус backend: реализовано до Phase 10. Документ `39_PHASE_7_KPI_SIGNALS_ACTION_ENGINE_BACKEND.md` остается контрактом; Phase 10 проверяет traceability, governed action path, permissions, audit и tenant isolation.

Цель: KPI definitions, evaluations, control signals, corrective actions, action engine, audit viewer.

Exit gate: KPI deviation превращается в governed action и audit.

Scenario planning после Phase 6 является готовой backend capability для будущих schedule/resource control signals. Сами KPI/control surfaces остаются scope Phase 7+ и не входят в Phase 5/6.

## Phase 8. Control surfaces builder

Статус backend: реализовано (PR #25, hotfix PR #26). Документ `40_PHASE_8_CONTROL_SURFACES_BUILDER_BACKEND.md` остается контрактом; Phase 10 проверяет versioning, archive/publish/rollback edge cases и action binding safety.

Цель: configurable management instruments, saved views, fields, filters, widgets, actions, preview/publish.

Exit gate: tenant admin публикует surface; обычный user работает с ней по правам.

## Phase 9. Closure and retrospectives

Статус backend: реализовано (PR #27, hotfix PR #30). Phase 10 проверяет immutable snapshot consistency, plan/fact metrics, retry/conflict semantics, template improvement audit и closure read models.

Цель: closure workflow, immutable snapshots, plan/fact, lessons, template improvement action.

Exit gate: closed project snapshot влияет на будущую оценку через governed improvement.

## Phase 10. Production hardening

Контракт: `41_PHASE_10_BACKEND_HARDENING.md`. Capability inventory: `docs/status/2026-05-25-backend-capability-inventory.md`.

Цель: security, privacy, performance, backup, deployment, self-hosted packaging, monitoring, docs for operators.

Exit gate: полный E2E контур проходит в release-like окружении.

Статус backend: в работе. Phase 10 включает не только cleanup, но и production readiness: backup/restore contract, self-hosted deployment docs, env/secrets validation, health/readiness, migration/runbook policy и performance/load smoke для ключевых read models.

## Phase G / Phase 11. Collaboration & Communications backend

Контракт: `42_PHASE_G_11_COLLABORATION_COMMUNICATIONS_BACKEND.md`.

Цель: сделать backend слой коммуникаций вокруг проектов, задач и CRM/opportunity: обсуждения, упоминания, unread/read state, pinned messages, notifications, meetings/calls, external video links, notes, recordings/files через Storage layer, decisions/action items и audit/security.

Exit gate: пользователь с доступом к project/task/opportunity может создавать обсуждения, сообщения, упоминания, meeting notes и action items; адресаты получают in-app notifications; недоступные entity не раскрываются; все write-flow permissioned/audited; external video links validated as metadata only; собственный WebRTC и frontend UI не входят.

### Phase G.2 / Phase 11.2. Communications realtime backend

Контракт: `43_PHASE_G_2_COMMUNICATIONS_REALTIME_BACKEND.md`.

Цель: расширить Phase G от external video links к безопасному backend control-plane для аудио/видео: call rooms, sessions, provider abstraction (`manual | jitsi | livekit`), short-lived join-token contract, participant state, call events, recordings via Storage layer, permissions and audit.

Exit gate: пользователь с доступом к parent project/task/opportunity может получить безопасный join contract, управляемый пользователь может создать комнату и session, участники меняют presence state, запись привязывается только через same-entity attachment, а audit не содержит provider secrets, tokens, storage keys или private internals.

## Phase 12. Calendar & Occupancy V2 backend

Контракт: `44_PHASE_12_CALENDAR_OCCUPANCY_V2_BACKEND.md`.

Цель: сделать единый backend source of truth для занятости сотрудника: planning assignments, reservations, absences, personal calendar busy events, meetings и calls должны сходиться в один minute-slot occupancy слой, который используют planning/resource load, tenant capacity и auto-solver.

Exit gate: personal busy event, meeting participant и call session уменьшают свободную capacity; private/busy-only metadata маскируется; auto-solver предпочитает no-overlap minute-slot proposal; все writes permissioned/audited; future Google/Microsoft/CalDAV sync boundary закреплен без server-side fetch.

Non-scope: frontend calendar UI, OAuth/webhook sync jobs, provider-side calendar creation, BI/export.

## Phase 13. Background Jobs Infrastructure

Контракт: `45_PHASE_12_BACKGROUND_JOBS_INFRASTRUCTURE.md`.

Цель: добавить общий backend foundation для scheduled/background jobs: durable DB-backed queue, persisted schedules, retries, cleanup jobs для archived files/assets, notification dispatch boundary, connector sync boundary, search projection rebuild, capacity cache warmup и observability trail.

Exit gate: self-hosted runtime может включить in-process worker без внешнего broker; tenant admin с правом `tenant.background_jobs.read/manage` видит и создает job runs через API; scheduled jobs enqueued idempotently; failed jobs retry with bounded backoff; storage cleanup purges only archived assets after retention; job events дают диагностику без provider secrets.

## Phase H / Phase 14. Documents & Knowledge Layer backend

Контракт: `44_PHASE_H_DOCUMENTS_KNOWLEDGE_LAYER_BACKEND.md`.

Цель: сделать tenant-scoped backend слой проектных документов и знаний поверх storage/collaboration: project documents, meeting minutes, decision log, action items, document attachments, версионирование и future approvals boundary.

Exit gate: пользователь с доступом к проекту может читать документы, решения и action items; пользователь с правом управления проектом может создавать документы/версии/решения/action items; meeting/document/decision ссылки проверяются на принадлежность проекту; вложения типа `document` идут через общий Storage layer; metadata-only search находит readable knowledge records; audit не хранит тело документов.

### Phase G.3 / Phase 15. Communications upgrade backend

Контракт: `44_PHASE_G_3_COMMUNICATIONS_UPGRADE_BACKEND.md`.

Цель: довести коммуникационный backend до рабочего уровня: общие workspace/team/project каналы, entity-scoped чаты для project/task/opportunity/client/contact, reactions/emoji, tenant sticker packs/imports через Storage, channel-scoped audio/video rooms, permissioned notifications и audit.

Exit gate: пользователь может вести общий чат и обсуждение в карточках проекта/задачи/CRM, отправлять сообщения, реакции и безопасные stickers, запускать channel/entity call room через provider control-plane, а backend доказывает tenant isolation, permissions, audit safety, storage safety и backward compatibility с Phase G/G.2.
