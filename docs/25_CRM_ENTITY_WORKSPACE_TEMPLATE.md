# 25. CRM entity workspace template

## Назначение

Этот документ задает единый шаблон экранов CRM-сущностей KISS PM: сделка, клиент, контакт, товар/услуга и будущие CRM-объекты. Шаблон нужен, чтобы новые страницы не превращались в разрозненные CRUD-таблицы и не копировали случайные паттерны из внешних CRM.

## Принцип

CRM-экран должен быть рабочим пространством сущности:

```txt
список / kanban
  -> карточка сущности
  -> контекстные действия
  -> коммуникации / задачи / системная лента
  -> audit evidence там, где действие меняет состояние
```

Нельзя добавлять видимый контрол, если за ним нет persisted сценария, disabled-state reason или явно описанного future scope.

## Общая структура карточки

Карточка любой CRM-сущности строится из блоков:

1. **Header**
   - breadcrumbs;
   - название сущности;
   - статус;
   - основные действия;
   - secondary меню только с реальными действиями.

2. **Metric / signal strip**
   - компактные числовые факты;
   - только данные, которые помогают принять решение;
   - без декоративных KPI.

3. **Main fact panel**
   - основные поля;
   - связи с другими сущностями;
   - кастомные поля tenant-а;
   - lifecycle/status strip, если у сущности есть стадии.

4. **Related entity cards**
   - клиент, контакт, проект, товарные позиции, документы;
   - карточки кликабельны, если у пользователя есть право на целевую сущность;
   - если связь еще не реализована, блок не должен выглядеть как доступное действие.

5. **Right activity workspace**
   - лента;
   - чат/комментарии;
   - задачи;
   - файлы;
   - audit tab только при наличии права и реального audit source.

6. **Governed action area**
   - действия, которые меняют бизнес-состояние;
   - обязательная проверка permissions на backend;
   - audit trail;
   - причина для необратимых или финальных переходов.

## Список сущностей

Список сущности должен иметь:

- поиск по реальным полям;
- понятные empty/loading/error states;
- строки, открывающие карточку сущности;
- inline actions рядом с объектом;
- отсутствие bulk selection, export, сортировки или фильтров, если они не реализованы end-to-end.

## Сделка

Сделка является входом в проектный контур. Ее карточка должна показывать:

- клиента;
- контакт;
- тип проекта;
- этап;
- плановые даты;
- стоимость контракта;
- норму часа;
- расчетные плановые часы;
- demand по должностям;
- resource feasibility;
- activity workspace;
- governed actions: активация проекта, закрытие/отклонение сделки.

## Клиент

Клиент является самостоятельной CRM-сущностью. Карточка должна показывать:

- статус;
- описание;
- контакты клиента;
- связанные сделки и проекты, когда соответствующие списки доступны;
- activity workspace как future block, если коммуникационная модель клиента еще не реализована.

## Контакт

Контакт является самостоятельной CRM-сущностью, но всегда связан с клиентом. Карточка должна показывать:

- клиента;
- роль у клиента;
- email, телефон, Telegram;
- статус;
- связанные сделки, когда связь доступна;
- переход к клиенту без дублирования CRUD клиента внутри карточки контакта.

## Товар/услуга

Товар/услуга является полноценной CRM-сущностью каталога, а не полем сделки.

Минимальная модель:

- `id`;
- `tenantId`;
- `name`;
- `sku`;
- `type`: `service` или `goods`;
- `unit`;
- `price`;
- `description`;
- `status`;
- `createdAt`;
- `updatedAt`.

В текущем baseline товар/услуга не привязывается к сделке, потому что еще нет модели состава сделки (`DealLineItem`). UI не должен показывать кнопку “добавить в сделку”, пока не появятся persisted line items, пересчет экономики и audit.

## Permission baseline

- чтение клиента: `tenant.clients.read`;
- управление клиентом: `tenant.clients.manage`;
- чтение контакта: `tenant.contacts.read`;
- управление контактом: `tenant.contacts.manage`;
- чтение товаров и услуг: `tenant.products.read`;
- управление товарами и услугами: `tenant.products.manage`;
- чтение сделки: `tenant.opportunities.read`;
- управление сделкой: `tenant.opportunities.manage`;
- audit evidence: `tenant.audit_events.read`.

## Следующий продуктовый срез

После появления самостоятельного каталога товаров/услуг следующий логичный slice:

```txt
DealLineItem
  -> выбор товара/услуги в сделке
  -> количество / цена / скидка / сумма
  -> пересчет экономики сделки
  -> audit action
  -> отображение товарных позиций в карточке сделки
```
