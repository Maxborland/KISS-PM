# 25. CRM Entity Workspace: шаблон карточек CRM-сущностей

## Статус

Этот документ фиксирует canonical layout для детальных экранов CRM-блока KISS PM.

Шаблон применяется ко всем карточкам CRM-сущностей:

- сделка;
- клиент;
- контакт;
- товар, услуга или продуктовая позиция;
- счет, документ или коммерческое предложение;
- будущие tenant-defined CRM-сущности, если они участвуют в проектном контуре.

Карточка CRM-сущности не является статичной формой просмотра. Это рабочее место вокруг объекта: пользователь видит состояние, связи, обязательные поля, следующие действия, коммуникации, задачи, историю и governed actions в одном контексте.

## Цель шаблона

Дать единый макет, по которому можно строить любой CRM detail screen без повторного проектирования с нуля.

Шаблон должен обеспечивать:

- одинаковую навигационную модель для всех CRM-сущностей;
- плотный рабочий интерфейс без landing-style декоративности;
- tenant-configurable поля, секции, стадии и действия;
- явную связь между карточкой CRM и проектным контуром KISS PM;
- отсутствие fake affordances: каждая кнопка, вкладка и меню либо работает, либо скрыто, либо явно disabled с причиной;
- проверяемые permission, audit и E2E-контракты для state-changing действий.

## Продуктовый принцип

CRM в KISS PM — это вход в управление проектами, ресурсной загрузкой и управленческим контролем.

Поэтому карточка CRM-сущности должна отвечать на вопросы:

1. Что это за объект?
2. В каком он состоянии?
3. Какие данные обязательны для следующего шага?
4. С чем и с кем он связан?
5. Что пользователь может сделать сейчас?
6. Что система уже сделала или зафиксировала?
7. Как это влияет на intake, resource feasibility, проект, KPI или audit?

## Обязательные зоны

### 1. Entity Header

Header дает пользователю быстрый ответ: где он находится, какой объект открыт и какое главное действие доступно.

Состав:

- breadcrumb: раздел -> сущность;
- название объекта;
- короткий статус или stage badge;
- ключевая metadata строка;
- primary action;
- secondary actions;
- overflow menu;
- индикаторы загрузки, сохранения или locked/final state.

Правила:

- на экране может быть только один визуально главный primary action;
- destructive actions не смешиваются с обычными действиями и требуют подтверждения;
- action menu не должен содержать будущие пункты без реализации;
- если действие недоступно, UI показывает disabled reason или скрывает действие по permission model.

Примеры primary actions:

| Сущность | Primary action |
|---|---|
| Сделка | `Создать проект`, `Проверить ресурсы`, `Закрыть сделку` в зависимости от состояния |
| Клиент | `Создать сделку` |
| Контакт | `Создать задачу` или `Создать сделку` |
| Товар/услуга | `Добавить в сделку` |
| Счет/документ | `Отправить`, `Согласовать`, `Сформировать PDF` |

### 2. Lifecycle / Stage Bar

Lifecycle bar показывает состояние объекта во времени.

Для сделки это pipeline stage. Для клиента это lifecycle/status. Для контакта это engagement/status. Для товара или услуги это статус доступности, публикации или архивности.

Правила:

- стадии и статусы берутся из tenant-настроек или системной lifecycle-модели, не hardcoded в React;
- текущий этап виден без чтения карточки;
- недоступные переходы имеют причину;
- переход состояния выполняется через application command;
- state-changing transition пишет audit;
- архивные или финальные состояния не исчезают из исторических карточек.

Минимальный contract:

```txt
currentState
availableTransitions[]
transitionBlockerReason | null
lastStateChangedAt
lastStateChangedBy
```

### 3. Context Tabs

Tabs разделяют поверхности работы, но не ломают контекст карточки.

Базовый набор:

- `Общие` — основные поля, readiness, связи и ключевые проверки;
- `Связи` — связанные сделки, клиенты, контакты, проекты, документы, товары;
- `Документы` — счета, КП, договоры, файлы;
- `Задачи` — задачи по объекту;
- `История` — audit/system events и изменения;
- `Еще` — только реально доступные дополнительные поверхности.

Правила:

- вкладка появляется только если у нее есть рабочая поверхность, данные или понятное empty state;
- будущая вкладка не должна выглядеть активной;
- вкладки должны deep-link или сохранять state, если пользователь возвращается к карточке;
- табы не заменяют права: backend permission check обязателен.

### 4. Entity Facts / Fields Pane

Левая часть карточки отвечает за структурированные данные объекта.

Типовые секции:

- summary section;
- обязательные поля;
- бизнес-поля сущности;
- custom sections;
- custom fields;
- связанные объекты;
- readiness/blockers;
- экономические или проектные показатели;
- system metadata.

Правила:

- секции и поля должны поддерживать tenant customization;
- обязательные поля показывают готовность и отсутствие данных;
- inline edit допустим только там, где он не ломает validation и audit;
- редактирование сложной секции открывает modal/sheet с validation;
- `Создать поле`, `Выбрать поле`, `Удалить раздел` доступны только в builder/admin mode;
- обычный пользователь не должен видеть технические настройки карточки, если он не редактирует шаблон.

Минимальный contract field section:

```txt
sectionId
label
order
visibilityRule
fields[]
isRequiredSection
canEdit
canConfigure
```

Минимальный contract field:

```txt
fieldId
tenantLabel
targetEntity
type
value
required
readonly
validation
source
```

### 5. Entity Activity Workspace

Правая часть карточки отвечает за живую работу вокруг объекта.

Состав:

- action composer;
- comments;
- tasks;
- communications;
- files/attachments;
- links;
- meetings/events;
- system events;
- audit context.

В KISS PM activity workspace должен поддерживать сценарий:

```txt
увидел состояние -> добавил действие/задачу/комментарий -> получил persisted event -> видишь историю и следующий шаг
```

Правила:

- комментарий, задача, файл, письмо, встреча и ссылка являются разными типами activity, а не декоративными вкладками;
- если activity type еще не реализован, пункт не показывается или disabled с причиной;
- audit не должен быть главным пользовательским табом, если пользователь решает рабочую задачу;
- audit events можно показывать в ленте как system events, а полную audit-поверхность оставить для админов;
- создание задачи или комментария должно сохраняться, переживать reload и иметь E2E-доказательство.

Минимальный activity contract:

```txt
activityId
entityType
entityId
type
body
status
authorId
assigneeUserId | null
dueDate | null
createdAt
updatedAt
```

### 6. Relations / Linked Objects

Связи показывают объект в бизнес-контексте.

Примеры:

- сделка -> клиент, контакт, тип проекта, будущий проект, документы, задачи;
- клиент -> контакты, сделки, проекты, счета, документы;
- контакт -> клиент, сделки, задачи, коммуникации;
- товар/услуга -> сделки, шаблоны проектов, цены, документы;
- счет/документ -> сделка, клиент, контакт, согласования, файлы.

Правила:

- связанные объекты должны быть кликабельными, если пользователь имеет право чтения;
- нет права чтения — показываем безопасный label или скрываем ссылку;
- relation cards не должны открывать dead route;
- удаление или архивирование связанного объекта не должно ломать историческую карточку.

## Entity-specific templates

### Сделка

Назначение: провести opportunity через intake к проекту.

Обязательные блоки:

- клиент;
- основной контакт;
- тип проекта;
- стадия сделки;
- плановые даты;
- стоимость контракта;
- плановая норма часа;
- расчет плановых часов;
- demand `должность + часы`;
- resource feasibility;
- readiness/blockers;
- project activation action;
- governed final actions: выиграна, отклонена, закрыта.

Primary actions:

- `Проверить ресурсы`;
- `Создать проект`;
- `Закрыть как выигранную`;
- `Отклонить сделку`.

Ключевой E2E:

```txt
создать сделку -> заполнить обязательные поля -> проверить ресурсы
  -> создать active project -> увидеть audit и связанную карточку проекта
```

### Клиент

Назначение: показать компанию или заказчика как источник сделок, контактов и проектов.

Обязательные блоки:

- тип клиента;
- статус;
- ответственный;
- контакты;
- активные сделки;
- активные/закрытые проекты;
- документы;
- коммуникации;
- история изменений.

Primary actions:

- `Создать сделку`;
- `Добавить контакт`;
- `Запланировать действие`;
- `Архивировать клиента` с подтверждением.

Ключевой E2E:

```txt
создать клиента -> добавить контакт -> создать сделку из клиента
  -> открыть сделку и увидеть связь с клиентом
```

### Контакт

Назначение: показать человека, его роль в клиенте и связанные рабочие действия.

Обязательные блоки:

- клиент;
- должность;
- роль в сделке или отношениях;
- email;
- телефон;
- связанные сделки;
- задачи;
- коммуникации;
- notes.

Primary actions:

- `Создать задачу`;
- `Создать сделку`;
- `Добавить коммуникацию`.

Ключевой E2E:

```txt
создать контакт у клиента -> создать сделку с этим контактом
  -> открыть карточку контакта и увидеть связанную сделку
```

### Товар / услуга / продуктовая позиция

Назначение: описать продаваемую или проектную позицию, которую можно связать со сделкой, документом или проектным шаблоном.

Обязательные блоки:

- название;
- тип;
- статус;
- цена или ставка;
- единица измерения;
- описание;
- применимость к типам проектов;
- связанные сделки;
- связанные шаблоны проектов;
- архивность.

Primary actions:

- `Добавить в сделку`;
- `Связать с шаблоном проекта`;
- `Архивировать`.

Ключевой E2E:

```txt
создать услугу -> добавить ее в сделку -> увидеть влияние на экономику сделки
```

### Счет / документ / коммерческое предложение

Назначение: сопровождать коммерческую и юридическую часть CRM-объекта.

Обязательные блоки:

- связанная сделка;
- клиент;
- контакт;
- статус согласования;
- сумма;
- файл или generated artifact;
- события отправки/согласования;
- история изменений.

Primary actions:

- `Сформировать`;
- `Отправить`;
- `Согласовать`;
- `Отменить` или `Архивировать` с подтверждением.

Ключевой E2E:

```txt
создать документ по сделке -> отправить/согласовать
  -> увидеть событие в activity workspace и audit
```

## Responsive behavior

### Desktop

Desktop layout использует две основные рабочие колонки:

- слева: fields/facts pane;
- справа: activity workspace.

Activity workspace может быть sticky, если высота экрана позволяет, но не должен создавать конфликт nested scroll.

### Tablet

Tablet layout допускает:

- верхний header и lifecycle на всю ширину;
- facts pane и activity workspace в две колонки, если ширина достаточна;
- activity workspace под facts pane, если ширина ограничена.

### Narrow viewport

На узком viewport:

- sidebar сворачивается;
- header actions переходят в compact layout;
- lifecycle bar становится scrollable или vertical stepper;
- facts sections идут одной колонкой;
- activity workspace становится отдельным блоком ниже или доступен через sticky tab;
- горизонтального overflow страницы быть не должно.

## UI quality rules

Обязательные правила:

- русский UI;
- плотность данных выше декоративности;
- понятные empty/loading/error states;
- focus states для keyboard navigation;
- disabled states с причиной;
- destructive actions через confirmation;
- no fake controls;
- no one-off цветовые хаки вне design tokens;
- icons из единой библиотеки;
- cards используются для смысловых объектов и секций, но не вкладываются бесконтрольно друг в друга;
- timeline и activity должны быть сканируемыми по времени, автору, типу и результату.

## Architecture boundaries

Карточка CRM-сущности должна собираться из повторяемых компонентов, а не из одного большого файла.

Рекомендуемые UI-модули:

```txt
CrmEntityPage
CrmEntityHeader
CrmLifecycleBar
CrmEntityTabs
CrmFieldsPane
CrmFieldSection
CrmRelationCard
CrmActivityWorkspace
CrmActivityComposer
CrmActivityTimeline
CrmEntityActionsMenu
```

Рекомендуемые application/domain границы:

```txt
CRM entity read model
field section/template resolver
lifecycle transition service
activity service
relation resolver
permission policy
audit writer
```

Правило зависимости:

```txt
UI -> query/application API -> application service -> domain/persistence
```

Domain logic не живет в React-компонентах.

## Permissions and audit

Минимальные permission surfaces:

- read entity;
- manage entity;
- manage lifecycle;
- manage fields/template;
- create activity;
- manage tasks;
- manage documents;
- view audit;
- perform governed action.

Все state-changing действия должны:

- проверять backend permission;
- валидировать текущий lifecycle state;
- выполнять изменение через application command;
- писать audit event;
- обновлять activity/system event, если событие важно пользователю;
- переживать reload.

## Anti-patterns

Запрещено:

- строить карточку как один большой React-файл;
- hardcode стадий, ролей, KPI или полей в UI;
- показывать пункты меню без backend-сценария;
- использовать `report` как основной доменный концепт CRM-карточки;
- делать audit основной пользовательской поверхностью вместо рабочего activity stream;
- скрывать проблему только UI-hiding без backend permission;
- делать связи текстом без кликабельного маршрута, если маршрут существует;
- создавать отдельные несовместимые layouts для сделки, клиента и контакта;
- переносить Bitrix naming как доменную модель KISS PM.

## Acceptance criteria для любой новой CRM-карточки

Карточка считается построенной по шаблону, если:

1. Есть deep link на карточку.
2. Header показывает название, статус, metadata и primary action.
3. Lifecycle/status отображается из модели, а не из hardcoded UI.
4. Есть facts/fields pane с обязательными и custom полями.
5. Связанные объекты кликабельны или имеют понятную причину недоступности.
6. Activity workspace показывает реальные persisted события, задачи или empty state.
7. State-changing actions проверяют permission и пишут audit.
8. Нет видимых fake controls.
9. Desktop и narrow viewport не имеют горизонтального overflow.
10. E2E доказывает основной user path карточки, включая reload.

## Implementation checklist

Перед реализацией новой CRM-карточки нужно ответить:

- какая сущность является source of truth;
- какие связи обязательны;
- какие поля системные, tenant-configurable и runtime custom;
- какой lifecycle используется;
- какие actions являются governed;
- какие permissions нужны;
- какие audit events создаются;
- какие activity types реально поддержаны;
- какие вкладки нужны сейчас, а какие не показываем до реализации;
- какой E2E доказывает ценность страницы.

## Решение

CRM-блок KISS PM использует единый pattern:

```txt
header + lifecycle + tabs + fields pane + activity workspace + relations + governed actions
```

Это основной layout для карточек сделки, клиента, контакта, товара/услуги, счета/документа и будущих CRM-сущностей.

## Runtime уточнения текущей реализации

## Activity contract

CRM activity является общей tenant-scoped моделью, а не частью только сделки:

```txt
CrmActivity
  -> tenantId
  -> entityType: opportunity | client | contact | product
  -> entityId
  -> type: comment | task | file
  -> title/body/status/dueDate/assignee/author
  -> fileUrl/fileSizeBytes/mimeType для file metadata
```

Контракт намеренно не включает binary storage: в текущем slice файл является persisted metadata/link. Любая запись активности создается только через backend action с entity-specific permission:

- `opportunity` -> `tenant.opportunities.read/manage`;
- `client` -> `tenant.clients.read/manage`;
- `contact` -> `tenant.contacts.read/manage`;
- `product` -> `tenant.products.read/manage`.

`fileUrl` принимает только абсолютные `http://` или `https://` ссылки. Относительные пути, `javascript:` и другие схемы не входят в текущий контракт: storage/connector layer проектируется отдельным срезом.

Плановый следующий слой для настоящих вложений и внешних источников описан в `26_PHASE_4_2_STORAGE_CONNECTOR_FOUNDATION.md`. После него `CrmActivity.type = file` должен ссылаться на `FileAsset` или `ExternalReference`, а не хранить только свободный URL.

Сделка в финальном статусе блокирует новые комментарии, задачи и файлы. Клиент, контакт и товар пока не имеют финального lifecycle-lock, поэтому их activity остается редактируемой при наличии manage-права.

## Inline-редактирование карточки

Карточка CRM-сущности должна быть рабочей поверхностью, а не только страницей просмотра. В левой колонке все бизнес-поля сущности, кроме расчетных и системных, редактируются inline:

- значение открывает компактный редактор по клику;
- изменение применяется только после явного подтверждения `Сохранить`;
- рядом всегда есть отмена без сохранения;
- validation error показывается у редактируемой строки;
- сохранение идет через тот же API/action contract, что и полноценная форма;
- backend permissions и audit не обходятся;
- расчетные поля показываются read-only.

Для сделки расчетными/read-only считаются `Плановые часы` и агрегированная `Потребность`, пока не появится отдельный редактор demand строк. Базовые поля сделки, tenant custom fields, поля связанной компании и контакта редактируются inline в рамках карточки.

## Список сущностей

Список сущности должен иметь:

- поиск по реальным полям;
- понятные empty/loading/error states;
- строки, открывающие карточку сущности;
- inline actions рядом с объектом;
- отсутствие bulk selection, export, сортировки или фильтров, если они не реализованы end-to-end.

## Сделка

Сделка является входом в проектный контур. Ее карточка должна показывать:

- клиента;
- контакт;
- тип проекта;
- этап;
- плановые даты;
- стоимость контракта;
- норму часа;
- расчетные плановые часы;
- demand по должностям;
- resource feasibility;
- activity workspace;
- governed actions: активация проекта, закрытие/отклонение сделки.

## Клиент

Клиент является самостоятельной CRM-сущностью. Карточка должна показывать:

- статус;
- описание;
- контакты клиента;
- связанные сделки и проекты, когда соответствующие списки доступны;
- persisted activity workspace с лентой, задачами и файлами.

## Контакт

Контакт является самостоятельной CRM-сущностью, но всегда связан с клиентом. Карточка должна показывать:

- клиента;
- роль у клиента;
- email, телефон, Telegram;
- статус;
- связанные сделки, когда связь доступна;
- persisted activity workspace с лентой, задачами и файлами;
- переход к клиенту без дублирования CRUD клиента внутри карточки контакта.

## Товар/услуга

Товар/услуга является полноценной CRM-сущностью каталога, а не полем сделки.

Минимальная модель:

- `id`;
- `tenantId`;
- `name`;
- `sku`;
- `type`: `service` или `goods`;
- `unit`;
- `price`;
- `description`;
- `status`;
- `createdAt`;
- `updatedAt`.

В текущем baseline товар/услуга не привязывается к сделке, потому что еще нет модели состава сделки (`DealLineItem`). UI не должен показывать кнопку “добавить в сделку”, пока не появятся persisted line items, пересчет экономики и audit.

Карточка товара/услуги использует тот же activity workspace, что сделка, клиент и контакт: комментарии, follow-up задачи и файловые ссылки сохраняются в `CrmActivity` с `entityType = product`.

## Permission baseline

- чтение клиента: `tenant.clients.read`;
- управление клиентом: `tenant.clients.manage`;
- чтение контакта: `tenant.contacts.read`;
- управление контактом: `tenant.contacts.manage`;
- чтение товаров и услуг: `tenant.products.read`;
- управление товарами и услугами: `tenant.products.manage`;
- чтение сделки: `tenant.opportunities.read`;
- управление сделкой: `tenant.opportunities.manage`;
- audit evidence: `tenant.audit_events.read`.

## Следующий продуктовый срез

После появления самостоятельного каталога товаров/услуг следующий логичный slice:

```txt
DealLineItem
  -> выбор товара/услуги в сделке
  -> количество / цена / скидка / сумма
  -> пересчет экономики сделки
  -> audit action
  -> отображение товарных позиций в карточке сделки
```
