# 01. Продуктовый контур

## Полный цикл

KISS PM строится вокруг замкнутого управленческого цикла.

```txt
Tenant setup
  -> users / roles / access
  -> CRM opportunity
  -> intake readiness
  -> demand forecast
  -> capacity feasibility
  -> project draft
  -> active project
  -> Gantt / WBS / baseline
  -> tasks / Kanban / assignments
  -> workload matrix / overload detection
  -> KPI evaluation / control signal
  -> governed management action
  -> audit trail
  -> closure snapshot
  -> retrospective insight
  -> improved templates
```

## Главные роли

- Собственник / директор: видит портфель, риски, деньги, сроки, перегрузы и стратегические действия.
- Операционный директор / resource manager: управляет емкостью, загрузкой, резервами и конфликтами.
- Project manager: ведет проект, план, задачи, стадии, отклонения и корректирующие действия.
- Project principal / stage lead: отвечает за качество, стадии, артефакты, принятие и эскалации.
- Исполнитель: работает через My Work / Kanban, видит свои роли в задачах и обновляет состояние.
- CRM / sales user: заводит opportunity и проверяет реализуемость до обещаний клиенту.
- Tenant admin: настраивает роли, права, поля, шаблоны, KPI, поверхности управления и интеграции.

## Ключевые пользовательские сценарии

### CRM -> проект

1. Пользователь видит opportunity.
2. Система проверяет полноту данных.
3. Пользователь выбирает или получает предложенный шаблон проекта.
4. Система оценивает спрос по ролям, стадиям и датам.
5. Система показывает емкость и риски.
6. Пользователь выбирает действие: запросить данные, зарезервировать емкость, создать черновик, отложить или принять риск.
7. После approval/won создается активный проект.

### Перегруз ресурса -> решение

1. Resource Load Control показывает перегруз.
2. Пользователь открывает затронутые задачи и проекты.
3. Система предлагает не одну магическую правку, а несколько сценариев решения: агрессивный, сбалансированный и устойчивый.
4. Пользователь сравнивает preview: дата финиша, сдвиг дедлайна, часы перегруза, затронутые люди, риски и требуемые approval.
5. Пользователь выбирает действие или сценарий: shift, split, reassign, reserve, escalate, accept risk или move deadline.
6. Система проверяет права и preconditions.
7. Действие применяется и записывается в audit.
8. Матрица и KPI пересчитываются.

### Сценарное планирование

KISS PM не должен искать единственный "идеальный" план. Для ресурсных конфликтов, горящих дедлайнов и изменений в проекте система должна предлагать несколько объяснимых управленческих компромиссов:

- агрессивный сценарий — раньше завершить или удержать жесткий дедлайн ценой видимых перегрузов и конфликтов;
- сбалансированный сценарий — сохранить дедлайн и минимизировать перегрузы;
- устойчивый сценарий — убрать перегрузы и показать честный сдвиг фактического дедлайна.

Сценарии являются proposal-layer поверх Gantt, scheduling engine и resource matrix. Они не применяются автоматически: пользователь видит plan delta, объяснение риска, требуемые права и audit trail перед применением.

### KPI deviation -> corrective action

1. KPI engine фиксирует отклонение.
2. Создается control signal.
3. KPI Control Surface показывает причину, источник и разрешенные действия.
4. Пользователь создает corrective action, открывает Gantt, запрашивает объяснение, эскалирует или принимает риск.
5. Действие проходит через action engine и audit.

### Закрытый проект -> улучшение шаблона

1. Проект закрывается через closure workflow.
2. Создается immutable snapshot.
3. Retrospective surface сравнивает план/факт.
4. Пользователь создает improvement action или обновляет шаблон/KPI threshold.
5. Будущие оценки становятся точнее.

## Принцип нескольких путей

Ключевое действие должно быть доступно из нескольких контекстов, если это естественно: создать задачу из Gantt, My Work, resource drilldown, KPI deviation или portfolio row. Но все пути должны вести к одной application command, одной проверке прав и одному audit trail.
