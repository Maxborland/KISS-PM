# 13. Глоссарий и антипаттерны

## Глоссарий

- Tenant — изолированная организация/рабочее пространство.
- Opportunity — коммерческая возможность, вход в проектный контур.
- ProjectIntake — проверка готовности opportunity к проекту.
- DemandProjection — прогноз трудозатрат и спроса на роли.
- CapacityAssessment — проверка, есть ли емкость выполнить работу.
- ProjectDraft — черновик проекта до активации.
- ActiveProject — проект в исполнении.
- Task — единая задача для всех views.
- Assignment — назначение работы на ресурс.
- ResourceLoadBucket — расчет загрузки за период.
- ResourceOverload — выявленный конфликт загрузки.
- PlanSnapshot — неизменяемый снимок плана, ресурсов, календарей, зависимостей и ограничений для расчета.
- PlanningScenario — предложенный вариант перепланирования с понятным профилем риска.
- PlanDelta — набор изменений, который можно применить к плану через governed action.
- Scenario Planning Engine — слой, который предлагает несколько объяснимых вариантов решения schedule/resource конфликта.
- KpiDefinition — определение KPI.
- KpiEvaluation — результат расчета KPI.
- ControlSignal — сигнал внимания или отклонение.
- ControlSurface — управленческая поверхность.
- ManagementInstrument — синоним control surface в продуктовой речи.
- ManagementAction — управленческое действие.
- CorrectiveAction — действие/задача для исправления отклонения.
- AuditEvent — запись факта изменения или решения.
- RetrospectiveSnapshot — снимок закрытого проекта.

## Запрещенные антипаттерны

### Report-first thinking

Плохо: экран просто показывает таблицу с проблемами.

Хорошо: экран показывает проблему, причину, affected entities, allowed actions, preview, audit и обновленное состояние.

### Hardcoded tenant process

Плохо: код знает стадии конкретной компании.

Хорошо: код знает stable system keys, а tenant labels и templates приходят из конфигурации.

### UI as domain logic

Плохо: React-компонент считает KPI или ресурсную загрузку.

Хорошо: UI вызывает API/query, domain engine считает, application service проверяет права.

### Action without audit

Плохо: кнопка меняет дату задачи.

Хорошо: кнопка вызывает command, command проверяет permission/preconditions, пишет audit и обновляет projections.

### Magic optimizer

Плохо: система автоматически перепланирует проект и показывает только новый результат.

Хорошо: система показывает несколько сценариев, объясняет plan delta, риски, перегрузы, сдвиг дедлайна и требует governed action перед применением.

### Disconnected modules

Плохо: CRM, Gantt, Kanban, resource matrix и KPI живут на разных моделях.

Хорошо: все views смотрят на единые project/task/assignment/KPI/action модели.

### Endless MVP

Плохо: добавлять экраны без конечного release journey.

Хорошо: каждая фаза ведет к полному доказанному управленческому контуру.
