# Поведенческий контракт: исправления persona journey

Дата: 2026-07-13
Ветка: `codex/persona-journey-fixes`
Источник: наблюдательный проход по персонам «руководитель delivery / владелец workspace» и «CRM-reader».

## Цель

Убрать найденные разрывы полного пути от входа в продукт до возврата и offboarding, не добавляя декоративных или неподключённых контролов. Все новые действия должны опираться на реальный API, локально сохранённый пользовательский контекст либо честно объяснять существующую операционную границу.

## Принятые допущения и границы

- Саморегистрация создаёт новый workspace. Пользователь, которого добавил администратор, входит по выданным ему учётным данным; self-service регистрация в чужой workspace без токена запрещена.
- В этом срезе нет email-провайдера приглашений и модели одноразовых invitation-токенов. Поэтому интерфейс не обещает письмо или ссылку-приглашение: он явно ведёт приглашённого пользователя на вход и объясняет, где получить данные.
- Запрос деактивации является реальным аудируемым событием, видимым администраторам в журнале аудита. Он не деактивирует пользователя автоматически и не обещает SLA.
- Удаление tenant, выгрузка всех данных и единственный системный владелец требуют отдельных data-retention и ownership-контрактов. До их появления UI не обещает эти операции. Текущий безопасный путь владельца — назначить другому пользователю профиль «Владелец», затем запросить обработку доступа.
- Возврат в последний контекст хранится локально для пары tenant/user и предлагается пользователю, а не выполняет неожиданный автоматический redirect.

## Наблюдаемое поведение и критерии приёмки

### A. Вход и создание workspace

1. Лендинг сохраняет основной CTA закрытой альфы и одновременно даёт существующему пользователю реальную ссылку входа, а новому — реальную ссылку создания workspace. URL приложения задаётся конфигурацией, не захардкожен на localhost.
2. Форма регистрации прямо говорит, что создаёт новый workspace, принимает его название и имя владельца.
3. API принимает `workspaceName`, валидирует его как однострочное непустое имя и сохраняет. Старый клиент без поля продолжает работать и получает имя, выведенное из имени владельца/email.
4. При успешной регистрации одной транзакцией создаются владелец, базовые статусы задач, базовый тип проекта и рабочая CRM-воронка с начальными/финальными стадиями. При ошибке нет частично созданного workspace.
5. После регистрации пользователь может добавить клиента и создать первую сделку без скрытого требования обратиться к администратору за типом проекта.
6. На пустом dashboard показан короткий setup-путь на реальных состояниях/ссылках; выполненные шаги отмечаются по данным API, а недоступные роли не видят обещаний действий без прав.

### B. Permission-aware интерфейс

7. Пользователь с `tenant.users.read`, но без `tenant.users.manage`, может читать список пользователей, но не видит активными создание, редактирование, деактивацию или реактивацию. Причина недоступности читаема.
8. Текущий пользователь не видит активной self-деактивации в таблице даже при `tenant.users.manage`; интерфейс объясняет безопасный следующий шаг. Серверные запреты остаются последней линией защиты.
9. Названия профилей доступа в списке и профиле человекочитаемы; внутренние tenant/access-profile ID не являются основным пользовательским текстом.
10. CRM-reader не получает инструкцию перетаскивать карточки и не получает текст «Нажмите Проверить», если проверка осуществимости ему запрещена. Текст называет требуемое право/следующего исполнителя.

### C. Agent, возврат и offboarding

11. При недоступном LLM-провайдере отправка всё равно добавляет сообщение пользователя в ленту, очищает поле, возвращает фокус и затем показывает честную квитанцию об отказе. Запрос к propose API не уходит.
12. После посещения сущностного/рабочего route dashboard предлагает «Продолжить работу» с последним доступным маршрутом; auth/admin/system routes и URL с чувствительными параметрами не сохраняются.
13. В профиле любой активный пользователь может отправить запрос деактивации. API требует сессию, записывает management audit event без изменения статуса и возвращает явную квитанцию. Повторный запрос идемпотентно создаёт новую датированную запись, а UI не заявляет, что доступ уже отключён.
14. Текст регистрации/входа объясняет путь приглашённого пользователя через реальную администраторскую учётную запись. Email-инвайт не изображается существующим.

## Ошибки и крайние случаи

- Невалидное название workspace возвращает `invalid_registration_payload`; пароль и email следуют текущим кодам ошибок.
- Если data source не поддерживает обязательные CRM-дефолты, регистрация отвечает `auth_not_configured`, а не создаёт tenant с неполным первым циклом.
- Ошибка setup-запроса не блокирует dashboard: показывается честный частичный статус и ссылки остаются permission-aware.
- Запрос деактивации без сессии — `401`; неактивный пользователь не может создать запрос; отказ аудита не выдаётся за успех.
- Последний route из другого пользователя/tenant не показывается.
