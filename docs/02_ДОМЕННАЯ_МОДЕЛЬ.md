# 02. Доменная модель

## Правило

Доменная модель должна быть универсальной. Tenant-специфичные названия, роли, стадии и KPI являются конфигурацией, а не кодом.

## Bounded contexts

1. Tenant и настройки.
2. Access control.
3. CRM intake.
4. Project lifecycle.
5. Work management.
6. Scheduling / Gantt.
7. Resource planning.
8. Scenario planning.
9. KPI control.
10. Control surfaces.
11. Action engine.
12. Audit.
13. Retrospectives.
14. Integrations.

## Ключевые сущности

### Tenant и пользователи

- `Tenant` — изолированная организация.
- `TenantUser` — пользователь внутри tenant.
- `AccessProfile` — набор прав и scope.
- `RoleTemplate` — конфигурируемая роль: project principal, stage lead, executor и т.д.
- `CustomFieldDefinition` — настраиваемое поле для сущности.

### CRM и приемка

- `ClientAccount` и `ClientContact`.
- `Opportunity` — коммерческая возможность.
- `ProjectIntake` — оценка готовности opportunity к проекту.
- `DemandProjection` — прогноз спроса по ролям/стадиям/датам.
- `CapacityAssessment` — проверка емкости.
- `IntakeDecision` — решение: запросить данные, создать draft, резервировать, отложить, принять риск.

### Проекты и задачи

- `Project` — проект tenant.
- `ProjectDraft` — черновик до активации. В текущей реализации это `Project` со статусом `draft`, а не отдельная таблица; решение закреплено в `docs/decisions/2026-05-19-project-draft-lifecycle-status.md`.
- `ProjectStage` — стадия проекта.
- `ProcessTemplate` и `StageTemplate` — шаблоны процесса.
- `Task` — единая задача для Gantt, Kanban, My Work, corrective actions и control surfaces.
- `TaskParticipant` — роль пользователя в задаче: executor, co-executor, requester, controller, approver, observer.
- `TaskDependency` — зависимость задач.
- `ProjectBaseline` — снимок плана.

### Ресурсы

- `ResourceProfile` — человек, роль, навыки, принадлежность к команде.
- `ResourceCalendar` — рабочее время и исключения.
- `Assignment` — назначение работы.
- `ResourceReservation` — резерв емкости.
- `ResourceLoadBucket` — рассчитанная загрузка за период.
- `ResourceOverload` — выявленный конфликт.

### Сценарное планирование

- `PlanSnapshot` — неизменяемый вход для расчета: задачи, зависимости, назначения, календари, отсутствия, capacity, baseline, deadlines и constraints.
- `PlanningScenario` — предложенный вариант перепланирования с профилем риска: aggressive, balanced или resilient.
- `PlanDelta` — набор изменений к плану: сдвиг, разделение, переназначение, резерв, принятие риска или сдвиг дедлайна.
- `ScenarioRiskProfile` — веса и ограничения, которые определяют, что считается дорогим: дата финиша, перегруз, нарушение дедлайна, дробление работы, смена исполнителя, skill mismatch.
- `ScenarioProposal` — explainable preview, который можно применить только через governed action.

### KPI, сигналы, действия

- `KpiDefinition` — описание KPI.
- `FormulaDefinition` — безопасная формула.
- `ThresholdRule` — пороги.
- `KpiEvaluation` — результат расчета.
- `ControlSignal` — сигнал внимания или отклонение.
- `CorrectiveAction` — корректирующее действие.
- `ManagementAction` — управленческое действие.
- `ActionExecution` — факт выполнения команды.
- `AuditEvent` — запись аудита.

## Инварианты

- Каждая tenant-owned сущность принадлежит ровно одному tenant.
- Проект может быть связан с CRM opportunity, но должен работать без внешней CRM.
- Нет отдельных моделей задач для Gantt, Kanban, отчетов и корректирующих действий.
- Control surface не мутирует состояние напрямую.
- Scenario planning не мутирует состояние напрямую: он возвращает proposal, а не новый project state.
- Scheduling engine остается детерминированным; scenario planning использует его как расчетный модуль, но не подменяет его.
- KPI evaluation traceable: формула, версия, источники, период, threshold, время расчета.
- Management action traceable: actor, source surface, source entity, command, before/after, result.
- Закрытые проекты используют snapshots, чтобы история не менялась молча.
