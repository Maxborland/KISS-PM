# 05. Роли, права и аудит

## Цель

Пользователь должен видеть и выполнять только те действия, которые разрешены его tenant, профилем доступа, ролью в проекте и scope.

## Модель доступа

- `AccessProfile` задает базовый набор прав.
- `Permission` описывает конкретную возможность: читать, создавать, менять, публиковать, выполнять action.
- `ScopeRule` ограничивает область: own, team, department, portfolio, all.
- Project/stage/task roles добавляют контекстные права.
- User override возможен, но должен быть аудируемым.

## Где проверяются права

Права проверяются в application services и action engine. UI может скрывать кнопки для удобства, но не является защитой.

## AuditEvent

Запись аудита создается для:

- выполнения management action;
- изменения tenant config;
- публикации шаблона или control surface;
- изменения KPI formula или threshold;
- изменения project lifecycle state;
- принятия риска или отклонения;
- override правила или stage gate;
- административных изменений пользователей/прав.

## Минимальный состав audit record

- tenantId;
- actorUserId;
- actorRole/context;
- actionType;
- sourceSurfaceId или sourceWorkflow;
- sourceEntity;
- input;
- before/after для значимых изменений;
- permission result;
- execution result;
- timestamp;
- correlationId.

## Правило отказа

Отказ в действии должен быть объяснимым. Для обычного пользователя — короткая причина. Для admin diagnostics — trace проверки прав.

## E2E-требование

Каждый state-changing management flow должен иметь минимум один позитивный и один негативный сценарий: authorized succeeds, unauthorized fails, audit exists.
