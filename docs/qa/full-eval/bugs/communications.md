## BUG-COMM-01 — Список каналов не обновляется после переименования (PATCH)

**Серьёзность:** низкая (косметика / стейл-стейт клиента).
**Где:** /communications/channels, `channels-surface` — сайдбар со списком каналов.

**Шаги:**
1. Войти admin@kiss-pm.local.
2. Открыть канал, нажать «Изменить», поменять «Название», нажать «Сохранить».
3. Наблюдать заголовок в детали (обновился) и пункт в сайдбаре слева (список каналов).

**Ожидание:** и деталь, и элемент списка в сайдбаре показывают новое название сразу после PATCH (без ручной перезагрузки страницы) — сервер вернул 200, данные актуальны.

**Факт:** деталь канала корректно показывает новое название («QA-Team-Channel-1-edited») и notice «Канал обновлён», но элемент в сайдбаре слева продолжает показывать старое название («QA-Team-Channel-1») до полной перезагрузки страницы (F5) — после reload список подтягивает верное название. Сетевой лог подтверждает: после `PATCH /communication-channels/:id` (200 OK) не было последующего `GET /communication-channels` (список не рефетчится), только точечный GET на конкретный канал.

**Доказательства:**
- Снапшот после PATCH: heading "QA-Team-Channel-1-edited" в детали, но `button "QA-Team-Channel-1 Команда"` в сайдбаре (тот же тик, до reload).
- Network log: `43 POST .../communication-channels → 201`, `49 PATCH .../communication-channels/channel-341d8f68-… → 200` — без парного `GET .../communication-channels` после PATCH.
- После `page.goto` (полный reload) — сайдбар показывает "QA-Team-Channel-1-edited" корректно.

## BUG-COMM-02 — Табы CommsFrame (Чат/Каналы/Звонки/Встречи/Уведомления) не являются навигацией

**Серьёзность:** средняя (нарушает принятый стандарт «каждый контрол — к реальному эндпоинту»; роуты физически существуют, но переключение между ними доступно только вводом URL).
**Где:** `apps/web/src/communications/ui/comms-frame.tsx:54-68`; все 5 роутов `/communications/*`.

**Шаги:**
1. Открыть /communications/channels под любой ролью.
2. Кликнуть по табу «Чат» (или «Звонки»/«Встречи»/«Уведомления») в верхней навигации внутри страницы.

**Ожидание:** переход на соответствующий роут (/communications/chat и т.д.) — вкладки должны быть кликабельной навигацией, т.к. все 5 роутов существуют и работают.

**Факт:** неактивные табы — это `<span>`/`<generic>` с title-подсказкой «Демо-прототип: переключение разделов появится в приложении», без роли `link`/`button`, без `href`, без обработчика клика. Переход между разделами возможен только прямым вводом URL в адресную строку. Подтверждено на трёх проверенных страницах (channels/chat/notifications) — на каждой активный (текущий) таб отрендерен как обычный `generic` без tooltip, а все остальные 4 таба — с этим tooltip-текстом.

**Доказательства:** accessibility-снапшот /communications/channels:
```
- navigation:
  - 'generic "Демо-прототип: переключение разделов появится в приложении"': Чат
  - generic: Каналы   # текущая страница — без tooltip, но тоже НЕ ссылка
  - 'generic "Демо-прототип: переключение разделов появится в приложении"': Звонки
  - 'generic "Демо-прототип: переключение разделов появится в приложении"': Встречи
  - 'generic "Демо-прототип: переключение разделов появится в приложении"': Уведомления
```
DOM-проверка (`document.querySelectorAll('a')`) на /communications/notifications не нашла ни одного `<a>` с текстом соответствующих разделов.

## BUG-COMM-03 — Индикатор «Сохранено» фиктивный (статичен, ни к чему не привязан)

**Серьёзность:** низкая-средняя (вводит в заблуждение — выглядит как реальный статус синхронизации).
**Где:** правый верхний угол панели вкладок на всех 5 роутах /communications/*.

**Шаги:** открыть любую страницу /communications/*, выполнить произвольные мутации (создание канала, редактирование, удаление участника, отправка сообщения, сохранение настроек уведомлений) и понаблюдать за индикатором «Сохранено» (зелёная точка + текст) в правом верхнем углу панели вкладок.

**Ожидание:** индикатор отражает реальное состояние сохранения (загрузка/сохранено/ошибка) — либо не показывается, если не привязан ни к чему.

**Факт:** индикатор — статичный `<span>` с текстом «Сохранено» и зелёной пульсирующей точкой, присутствует всегда, независимо от состояния: до любых действий, во время открытых диалогов, сразу после успешных и после неудачных (RBAC-отклонённых) мутаций — ни разу не изменился за ~15 разных действий в ходе прогона (создание/правка/удаление канала и участников, отправка сообщений, PUT настроек уведомлений, отклонённое создание под read-ролью).

**Доказательства:** DOM-инспекция элемента:
```html
<span class="ml-auto hidden items-center gap-1.5 pr-2 text-[length:var(--text-sm)] text-[var(--success-text)] md:flex">
  <span class="v4-pulse size-1.5 rounded-full bg-[var(--success)]"></span>Сохранено
</span>
```
Нет `data-state`/условного рендера по факту мутации — элемент безусловен в разметке.

## BUG-COMM-04 — Переход из уведомления к источнику не реализован (route — плоский текст)

**Серьёзность:** средняя (заявленный в контракте UX-критерий «клик по уведомлению ведёт на сущность» невыполним).
**Где:** /communications/notifications, `notifications-surface.tsx:200`. Позиция инвентаря COMM-031.

**Шаги:**
1. Открыть /communications/notifications под admin.
2. Найти уведомление «Риск срыва срока» с указанным route `/projects/project-vektor-portal`.
3. Кликнуть по этой строке / попытаться перейти по ней.

**Ожидание:** клик ведёт на `/projects/project-vektor-portal` (или как минимум элемент — интерактивная ссылка).

**Факт:** `/projects/project-vektor-portal` отрендерен как обычный `<paragraph>` (не `<a>`, не кнопка), без onClick. Клика не существует — переход физически невозможен.

**Доказательства:**

## BUG-COMM-05 — Отметка уведомления прочитанным не идемпотентна

**Серьёзность:** низкая-средняя (искажение метаданных «прочитано когда», но без потери функциональности).
**Где:** `POST /api/workspace/notifications/:id/read`. Позиция COMM-029.

**Шаги (через fetch с cookie-сессией admin, `x-kiss-pm-action: same-origin`):**
1. `POST /api/workspace/notifications/notification-alpha-vektor-deadline/read` → зафиксировать `readAt`.
2. Через ~1.2 сек повторить тот же `POST` на то же уведомление.
3. Сравнить `readAt` в обоих ответах.

**Ожидание:** повторный вызов — no-op (readAt не меняется), т.к. уведомление уже прочитано.

**Факт:** `readAt` каждый раз перезаписывается текущим временем:
- 1-й вызов: `"lastReadAt/readAt": "2026-07-04T15:37:37.202Z"` (первый тест) / повтор в другом прогоне: `"2026-07-04T15:37:38.212Z"`
- 2-й вызов (то же уведомление, ~1.2 сек спустя): `"2026-07-04T15:37:38.423Z"`

Оба ответа — 200 OK, оба раза `readAt` разный (растёт с каждым вызовом) — подтверждает неидемпотентность.

**Доказательства:** прямой вывод `fetch`-скрипта:
```json
{"id":"notification-alpha-vektor-deadline","firstReadAt":"2026-07-04T15:37:37.202Z","secondReadAt":"2026-07-04T15:37:38.423Z","status1":200,"status2":200}
```

## BUG-COMM-06 — Unpin-роута нет: закрепление сообщения в чате необратимо

**Серьёзность:** средняя (отсутствующая функциональность — пользователь не может отменить свое же действие; на боевом контракте это постоянная деградация UX, а не просто edge-case).
**Где:** `POST/DELETE /api/workspace/conversations/:id/messages/:mid/pin`. Позиция COMM-013.

**Шаги (через fetch с cookie-сессией admin):**
1. Отправить сообщение в реальную беседу проекта (`conversation-922451b7-…`, `project-12f33619-…`).
2. `POST /conversations/:id/messages/:mid/pin` → закрепить.
3. `DELETE /conversations/:id/messages/:mid/pin` → попытаться снять закрепление.

**Ожидание:** либо DELETE поддерживается и снимает pin (200), либо это осознанное ограничение, зафиксированное в UI (кнопка «Открепить» скрыта/недоступна) — сейчас же UI показывает пункт «Закрепить» без внятного объяснения необратимости.

**Факт:** `POST .../pin` → 200 OK, сообщение закреплено. `DELETE .../pin` на тот же id → **404 Not Found** (пустое тело). Роута для снятия закрепления не существует на сервере в принципе — закрепление сообщения через API необратимо.

**Доказательства:**
```json
{"pinStatus":200,"pinBody":"{\"message\":{\"id\":\"message-8f437c5a-…\",…}",
 "unpinStatus":404,"unpinBody":"{}"}
```
