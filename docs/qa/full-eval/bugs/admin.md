## BUG-ADM-01 — Сырой код 403-ответа вместо RU-текста + полный каркас админки рендерится не-админам

**Серьёзность:** Высокая (UX/безопасность-восприятие, не сама авторизация)

**Где:** все 4 страницы `/admin/users|roles|security|audit`; `apps/web/src/admin/ui/admin-bits.tsx` (RU-карта `adminErr`); API — все GET-ручки `workspaceUserRoutes.ts`, `accessRoleRoutes.ts`, `workspaceConfigRoutes.ts`, `auditRoutes.ts`.

**Шаги:**
1. Войти как `plan-reader-no-resources@kiss-pm.local` (не-админ).
2. Перейти по прямому URL на `/admin/users`, `/admin/roles`, `/admin/security`, `/admin/audit`.

**Ожидание:** понятное состояние «нет прав» на русском языке; в идеале — страница вообще не рендерит админ-каркас для не-админа.

**Факт:**
- Полный каркас админки (сайдбар, шапка, табы, заголовок раздела) рендерится для не-админа — гварда на уровне web нет (только по коду, не по UI).
- Все 4 GET-ручки корректно отвечают `403 {"error":"permission_missing"}` (проверено сетевым логом), но UI выводит блок «Не удалось загрузить» с текстом **`permission_missing`** — сырой код вместо русского объяснения, потому что причина не входит в RU-карту `adminErr`.
- Для `resource-reader@kiss-pm.local` — то же самое (не отдельно перепроверял все 4 страницы, но код идентичен).
- Без сессии (простой curl на API): `401`, не `500` — корректно.
- Мутации (POST users, PUT security-policy) под non-admin: тоже `403 permission_missing` — авторизация НЕ сломана, страдает только текст ошибки.

**Доказательства:** скриншот `bug-adm-01.png` (E:\KISS-PM\bug-adm-01.png) — страница `/admin/users` под `plan-reader-no-resources`, видно полный каркас + текст `permission_missing`. Сетевой лог: 403 на `/api/workspace/users`, `/api/workspace/access-roles`, `/api/workspace/positions`, `/api/workspace/permission-catalog`, `/api/tenant/current/audit-events`.

**Класс:** аналогичен известному BUG-SHELL-06 (сырые коды вместо RU), применительно к разделу admin. Авторизация как таковая работает верно на API — это баг отображения + отсутствия web-гварда (уже зафиксировано в inventory замечание №4).

## BUG-ADM-02 — Обнуление прав роли, назначенной пользователям, сохраняется без предупреждения (реальная потеря доступа)

**Серьёзность:** Критическая

**Где:** `/admin/roles`, диалог «Изменить роль» (`apps/web/src/admin/*roles-surface*`), `PATCH /api/workspace/access-roles/:roleId`.

**Шаги:**
1. Войти как admin (tenant-alpha).
2. Открыть «Изменить права» для роли «Наблюдатель ресурсов» (`access-profile-resource-reader`) — назначена 5 пользователям, включая seed-аккаунт `resource-reader@kiss-pm.local`.
3. Снять ВСЕ чекбоксы (0 прав).
4. Нажать «Сохранить».

**Ожидание:** предупреждение о том, что роль назначена N пользователям и они потеряют доступ; либо явное двойное подтверждение перед сохранением 0 прав для занятой роли.

**Факт:** сохранение проходит мгновенно без единого предупреждения. Проверено практически: после сохранения `GET /api/auth/me` под `resource-reader@kiss-pm.local` вернул `"permissions": []` — реальная, подтверждённая потеря доступа для всех 5 назначенных пользователей, без возможности отмены кроме как вручную вернуть права через API/UI.

**Важно:** для проверки роль временно была лишена прав, эффект подтверждён, затем права восстановлены обратно (`tenant.project_resources.read`) — сайд-эффект на seed-аккаунт устранён.

**Доказательства:** сетевой лог `PATCH /api/workspace/access-roles/access-profile-resource-reader` с `permissions: []` → `200 OK`, ответ `{"accessRole":{...,"permissions":[]}}"`; последующий `GET /api/auth/me` под resource-reader → `permissions: []`.

## BUG-ADM-03 — Настройки безопасности (2FA, SSO, whitelist доменов) сохраняются, аудируются, но нигде не enforce'ятся

**Серьёзность:** Критическая (фейковый security-контрол)

**Где:** `/admin/security`, `PUT /api/tenant/current/security-policy`, вход `POST /api/auth/login`, создание пользователя `POST /api/workspace/users`.

**Шаги и факт:**
1. **2FA:** `PUT security-policy {twoFactorRequired: true}` → `200`. Логаут + `POST /api/auth/login` с email/паролем `resource-reader@kiss-pm.local` → `200`, сессия выдана сразу, без единого запроса второго фактора.
2. **SSO/SAML:** то же с `ssoSamlEnabled: true` → логин по email/паролю всё равно проходит `200`, никакого редиректа на SAML-провайдера.
3. **Whitelist доменов:** `domainAllowlist: ["kiss-pm.local"]` сохранён (`200`), затем `POST /api/workspace/users` с email `qa-adm-outsider2@notallowed.com` (домен вне списка) → `201 Created`, пользователь создан без ограничений.

**Ожидание:** включённая политика должна реально влиять на вход/создание пользователей, либо в UI должна быть явная пометка «настройка не применяется» (честная prototype-маркировка).

**Факт:** все три настройки — чисто декоративные: сохраняются в БД, попадают в аудит (`workspace.security_policy.updated`), но ни одна не влияет на поведение системы.

**Доказательства:** серия fetch-вызовов из консоли браузера (см. сетевые логи сессии): PUT security-policy → 200, затем login/create без каких-либо преград.

**Уборка:** политика возвращена к дефолту (`twoFactorRequired:false, ssoSamlEnabled:false, domainAllowlist:[]`) после проверки.

## BUG-ADM-04 — Удаление роли без подтверждения (один клик = необратимое действие)

**Серьёзность:** Средняя-высокая (UX/деструктив)

**Где:** `/admin/roles`, кнопка «Удалить роль» (`roles-surface.tsx:~103`).

**Шаги:**
1. Создать тестовую роль без назначений.
2. Нажать «Удалить роль».

**Факт:** `DELETE /api/workspace/access-roles/:id` уходит немедленно по одному клику, роль исчезает из таблицы без единого confirm-диалога. Единственная защита от промаха — `title`-подсказка на кнопке для НАЗНАЧЕННЫХ ролей («Назначена пользователям — удаление отклонится»), но сама кнопка активна и для свободных ролей удаление необратимо мгновенно.

**Ожидание:** confirm-диалог перед необратимым удалением роли.

**Доказательства:** сетевой лог `DELETE /api/workspace/access-roles/role-qa-zero-perms-role → 200 OK` сразу по клику, без модального окна между кликом и запросом.

**Примечание:** для НАЗНАЧЕННОЙ роли API корректно отклоняет удаление (`409 access_role_assigned`, RU-текст «Отклонено: Роль назначена пользователям — сначала переназначьте их» — показан верно), т.е. данные не теряются в этом случае. Баг — про отсутствие confirm на уровне UI в принципе, а не про потерю данных для занятых ролей.

## BUG-ADM-05 — RU-метки audit-событий для ролей никогда не применяются (несовпадение ключей)

**Серьёзность:** Средняя

**Где:** `apps/web/src/admin/ui/admin-bits.tsx`, константа `AUDIT_ACTION_LABEL`; фактические actionType из `apps/api/src/accessRoleRoutes.ts`.

**Факт:** в карте RU-меток определены ключи `"access_role.created"`, `"access_role.update"`, `"access_role.deleted"`, но API реально пишет в аудит события с actionType `"tenant.access_profile.created"`, `"tenant.access_profile.updated"`, `"tenant.access_profile.deleted"` (см. `accessRoleRoutes.ts:125,269,358`). Ключи никогда не совпадают → `auditActionLabel()` всегда возвращает `actionType ?? code`, то есть **все** события жизненного цикла ролей (создание/правка/удаление) в журнале аудита показывают сырой код дважды (и как «метку», и как код), например:

```
tenant.access_profile.updated   tenant.access_profile.updated
```

вместо ожидаемого «Роль изменена» / «Роль создана» / «Роль удалена».

**Ожидание:** правки ролей отображаются понятным русским текстом, как это уже сделано для пользователей (`workspace.user.created` → «Пользователь создан» и т.п. — это работает корректно).

**Доказательства:** снимок таблицы `/admin/audit` — строки `tenant.access_profile.updated`, `tenant.access_profile.created`, `tenant.access_profile.deleted` показаны без перевода, при этом соседние `workspace.user.created/updated` показаны переведёнными «Пользователь создан» / «Пользователь изменён». Прочитан код `admin-bits.tsx:68-85` — карта содержит явно устаревшие/неверные ключи (похоже на рефакторинг с `access_role.*` на `tenant.access_profile.*`, метки не обновили).

**Примечание (не новый баг, для полноты):** отдельно подтверждено, что *_denied события (`workspace.user.read_denied`, `tenant.access_profile.read_denied`, `workspace.security_policy.update_denied` и т.д.) и события `communications.*`/`collaboration.*` тоже показываются сырыми кодами — это соответствует документированному в inventory поведению «неизвестный actionType → сырой код» (ADM-021.2, не отдельный баг). BUG-ADM-05 выделен отдельно, потому что для ролей перевод СУЩЕСТВУЕТ в коде, но никогда не срабатывает из-за опечатки в ключах — это конкретный, чинибельный дефект, а не общий пробел покрытия.
