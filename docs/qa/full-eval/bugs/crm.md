# Баги: CRM

Область: `/crm/clients`, `/crm/contacts`, `/crm/deals`, `/crm/deals/[id]`, `/crm/products`.
Стенд: web http://127.0.0.1:3000 → API http://127.0.0.1:4020, реальный Postgres seed, роль admin (если не указано иное).

Известный общий баг (не дублируется здесь, только фиксируется факт): на всех 5 CRM-страницах виден прототип-баннер «Прототип … Данные in-memory» и декоративные табы CrmFrame с тултипом «Демо-прототип» — это BUG-014 сводного лога.

## BUG-CRM-01 — Создание продукта с дублирующимся SKU роняет сервер в 500 вместо конфликта — 🟢 ИСПРАВЛЕН

- **Статус:** 🟢 исправлен и покрыт регрессом (фаза 4, 2026-07-05).
- **Фикс:** `apps/api/src/crmRoutes.ts` — хелпер `productUniqueConflict()` ловит Postgres 23505 по индексам `products_tenant_id_sku_uidx` / `products_tenant_id_name_uidx`; POST-хендлер продукта возвращает 409 `product_sku_taken` / `product_name_taken` вместо 500 (house-паттерн, как `isCredentialEmailConflict`).
- **Регресс:** `apps/api/src/crmRoutes.db.test.ts` → «returns 409 (not 500) when creating a product with a duplicate SKU or name» — зелёный; весь CRM-сьют (7 тестов) проходит.
- **Фаза-5 live-проверка (:4020, 2026-07-05):** новый продукт `201`; повтор SKU → `409 product_sku_taken`; повтор имени → `409 product_name_taken`. End-to-end подтверждён.
- **Осталось (не в этом фиксе):** RU-строки для кодов `product_sku_taken`/`product_name_taken` в web-словаре (кластер F); PATCH продукта — тот же класс, не покрыт.
- **Серьёзность:** high
- **Где:** `/crm/products`, модалка «Новый продукт», `POST /api/workspace/products`
- **Шаги:**
  1. Открыть «Продукт», указать название, цену, SKU = `KISS-AUDIT` (уже занят продуктом «Аудит проектного контура»).
  2. Нажать «Создать».
- **Ожидание:** конфликт по уникальности SKU должен вернуть предсказуемый 409/422 с RU-сообщением (или, если уникальность не предполагается, продукт должен создаться 201).
- **Факт:** сервер отвечает `500 Internal Server Error`, UI показывает «Отклонено: internal_error». С уникальным SKU тот же запрос отрабатывает 201. Значит на дублирующемся SKU падает необработанное исключение (похоже на нарушение unique-констрейнта в БД, не пойманное в слое команды).
- **Доказательства:** network — `POST /api/workspace/products` → 500 (SKU `KISS-AUDIT`), затем `POST /api/workspace/products` → 201 (SKU `QA-SKU-UNIQUE-001`, тот же остальной payload).

## BUG-CRM-03 — Селект «Стадия» в карточке сделки не работает для сделок без pipelineId (легаси)

- **Серьёзность:** high
- **Где:** `/crm/deals/[id]`, блок «Параметры сделки» → комбобокс «Стадия»
- **Шаги:**
  1. Открыть карточку любой сделки из исходного сида (кроме той, что уже была перетащена в канбане) — например `opportunity-gorset-expansion` или `opportunity-vektor-audit`.
  2. Посмотреть на комбобокс «Стадия».
- **Ожидание:** select со всеми стадиями активной воронки, с выбранной текущей стадией (как в критериях CRM-020).
- **Факт:** select содержит **ровно один** `<option>` со значением, равным текущему `stageId`, но текстом **«— без стадии —»** (неверная подпись) — реальное имя стадии («Согласование», «Квалификация» и т.д.) нигде не отображается, и сменить стадию через этот select невозможно в принципе (других опций нет). Корневая причина: у этих opportunity `pipelineId: null` (сид без унификации воронок) — код фильтрует опции по `opportunity.pipelineId`, а не по `stage.pipelineId`/факту принадлежности стадии активной воронке. Как только сделка проходит через `PATCH /:id/stage` (например drag&drop в канбане), её `pipelineId` проставляется, и select начинает работать нормально (проверено на `opportunity-romashka-support`).
- **Доказательства:** `browser_evaluate` дамп `document.querySelectorAll('select')` — для `opportunity-gorset-expansion`: `options: ["— без стадии —"]`, `value: "deal-stage-negotiation"`; для той же сделки через API `pipelineId: null`. После смены pipelineId на `tenant-alpha-pipeline-default` (`opportunity-romashka-support`) select возвращает все 4 стадии с верными подписями. Скриншот `screenshots/bug-crm-03-stage-select-broken.png`.
- **Связано с:** пункт «Замечания по коду» №6 и №13 инвентаря (легаси-стадии/pipelineId).

## BUG-CRM-04 — Поля «Старт»/«Финиш» в карточке сделки никогда не заполняются

- **Серьёзность:** medium
- **Где:** `/crm/deals/[id]`, блок «Параметры сделки» → `input[type=date]` «Старт»/«Финиш»
- **Шаги:** открыть карточку любой сделки (проверено на 3 разных: `opportunity-gorset-expansion`, `opportunity-vektor-audit`, `opportunity-romashka-support`, включая созданную через API с валидными датами).
- **Ожидание:** поля показывают `plannedStart`/`plannedFinish` из API (например `2026-07-01`).
- **Факт:** оба поля всегда пустые, несмотря на то что API отдаёт корректные ISO-даты (`"plannedStart": "2026-07-01T00:00:00.000Z"`). Похоже, значение передаётся в `<input type="date">` как полная ISO-строка с временем вместо `YYYY-MM-DD`, из-за чего браузер отбрасывает его как невалидное. Подтверждающий сайд-эффект: прямой PATCH с такой же ISO-строкой в `plannedStart` возвращает `400 invalid_planned_dates`; с `.slice(0,10)` — проходит. Т.е. если пользователь отредактирует любое другое поле и нажмёт «Сохранить» не трогая даты, есть риск отправки некорректного формата дат (полноценно не проверено из-за форс-мажора: PATCH на уже закрытых тестовых сделках блокируется локом раньше, чем дошло бы до дат).
- **Доказательства:** `browser_evaluate` — `Array.from(document.querySelectorAll('input[type=date]')).map(i=>i.value)` → `["", ""]` на всех проверенных сделках; сравнение с `GET /api/workspace/opportunities` показывает непустые даты в API.

## BUG-CRM-05 — Создание сделки через UI полностью не работает (500) из-за захардкоженной несуществующей позиции спроса

- **Серьёзность:** critical
- **Где:** `/crm/deals`, модалка «Новая сделка» (`CreateDealDialog`, `deals-surface.tsx:404-416`), `POST /api/workspace/opportunities`
- **Шаги:**
  1. Открыть «Сделка», заполнить название/клиента/контакт/стадию (форма валидна, кнопка «Создать» активна).
  2. Нажать «Создать».
- **Ожидание:** сделка создаётся (201), появляется в канбане.
- **Факт:** сервер отвечает `500 Internal Server Error` при **каждой** попытке (воспроизведено дважды подряд). Причина: `CreateDealDialog` жёстко прописывает строку спроса с `positionId: "backend"`, а в позициях тенанта (`GET /api/workspace/positions`) такого id нет вовсе (`position-analyst`, `position-engineer`, `position-project-manager`, `position-lead`) — нарушение FK/ссылочной целостности не перехватывается и всплывает как необработанный 500 вместо понятной 400-ошибки валидации. Подтверждено: тот же `POST /api/workspace/opportunities` с `positionId: "position-engineer"` (валидный) отрабатывает 201 без ошибок.
- **Итог:** **создание сделки через UI полностью заблокировано** на этом тенанте для любого пользователя (admin/engineer) — основной CRM-флоу мёртв на старте.
- **Доказательства:** network — 2× `POST /api/workspace/opportunities` → 500 при создании через диалог; затем ручной `POST` с `positionId: "position-engineer"` → 201.

## BUG-CRM-08 — money() на канбане/прогнозе показывает «0 тыс ₽» для сумм < 1000 ₽ (и грубо округляет до тыс. в остальных случаях)

- **Серьёзность:** low
- **Где:** `/crm/deals` — канбан-карточки и таблица режима «Прогноз» (`apps/web/src/crm/deals/deals-surface.tsx:22`, локальная функция `money()`)
- **Шаги:** создать сделку с `contractValue: 100` (QA Очень Мелкая Сумма) и `contractValue: 500` (QA Мелкая Сумма), посмотреть карточку в канбане и строку «Новая» в «Прогнозе».
- **Ожидание:** сумма 100 ₽ должна быть видна как «100 ₽» (или хотя бы не читаться как «ноль»); аналогично для взвешенного значения.
- **Факт:** карточка «QA Очень Мелкая Сумма» (100 ₽) показывает **«0 тыс ₽»** — выглядит как нулевая сделка. Карточка «QA Мелкая Сумма» (500 ₽) показывает «1 тыс ₽» (Math.round(0.5)→1) — тоже искажение, просто в другую сторону. В таблице «Прогноз» по стадии «Новая» (2 сделки, 100+500=600 ₽) сумма показана как «1 тыс ₽», а взвешенное значение (300 ₽) — как «0 тыс ₽». Общий `crm-bits.money()` (используется в других местах CRM) содержит ветку для `v < 1000`, локальный `money()` в `deals-surface.tsx` — нет.
- **Доказательства:** снапшот канбана — карточки `opportunity-765c685a-...` (100 ₽ → «0 тыс ₽») и `opportunity-0f94a399-...` (500 ₽ → «1 тыс ₽»); таблица «Прогноз», строка «Новая»: «1 тыс ₽» / «0 тыс ₽».

## BUG-CRM-09 — Кнопка создания сущности не скрывается/не блокируется при отсутствии прав (RBAC UI-leak)

- **Серьёзность:** medium
- **Где:** `/crm/products` (проверено), шапка страницы, кнопка «Продукт»; вероятно системно на всех 5 CRM-роутах
- **Роль:** `plan-reader-no-resources@kiss-pm.local` (PR) — нет CRM-прав вовсе
- **Шаги:**
  1. Войти как PR, открыть `/crm/products` → корректно показан forbidden-экран «Доступ к продуктам ограничен».
  2. Заметить, что кнопка «Продукт» в шапке страницы осталась активной (не disabled, не скрыта).
  3. Нажать «Продукт» → модалка «Новый продукт» открывается полностью, форма заполняется и сабмитится.
- **Ожидание:** при forbidden-состоянии раздела кнопка создания должна быть скрыта или задизейблена — по крайней мере не открывать полноценную форму.
- **Факт:** кнопка кликабельна, модалка открывается, форма заполняется и сабмитится как обычно. Сервер корректно отклоняет POST (`403 permission_missing`, UI показывает «Отклонено: permission_missing») — **эксплуатации нет** (защита есть на API), но UI вводит пользователя в заблуждение и не мешает бессмысленному вводу данных.
- **Доказательства:** network — `GET /api/workspace/products` → 403 (весь Promise.all), затем `POST /api/workspace/products` → 403 с той же сессией; снапшот показывает одновременно forbidden-текст в `<main>` и открытый диалог создания.

## Проверенные кандидаты, НЕ подтвердившиеся как баг (для протокола)

- **`javascript:`-URL в файловой активности (`href={a.fileUrl}`):** реальный API (`POST /crm/opportunity/:id/files`) валидирует схему через `parseExternalReferenceUrl` (`apps/api/src/attachmentValidation.ts`) — принимает только `http:`/`https:`, отклоняет `javascript:` с `400 file_url_invalid`. Проверено прямым запросом. Уязвимость есть только в неиспользуемом в проде mock-бэкенде (Storybook) — на боевом контуре не воспроизводится.
- **XSS в свободных текстовых полях** (название клиента `<script>…</script>`, должность контакта `<b>bold</b>`, тело комментария `<img onerror=…>`): везде рендерится как текст (React экранирует), исполнения скриптов не зафиксировано.
- **Смена стадии назад (канбан drag&drop):** корректно блокируется `409 transition_not_allowed` с RU-сообщением, карточка остаётся на месте — работает как задумано.
- **Блокировка после финализации/активации (locked-режим):** прямой `PATCH /opportunities/:id` на закрытую сделку → `409 opportunity_update_locked`; повторная финализация → `409 opportunity_final_action_locked`; активация при конфликте без обоснования риска → `400`; всё соответствует критериям.
