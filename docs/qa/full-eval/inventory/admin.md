# Инвентарь: Админка

Full Product Evaluation Loop — область «Администрирование».
Роуты: `/admin` (redirect), `/admin/users`, `/admin/roles`, `/admin/security`, `/admin/audit`.

Источники: `apps/web/src/app/admin/**` (страницы), `apps/web/src/admin/**` (surface-компоненты, `admin-client.ts`, `use-admin.ts`, `use-security-policy.ts`, `use-audit-events.ts`, `admin-frame.tsx`, `admin-bits.tsx`), API: `apps/api/src/workspaceUserRoutes.ts`, `accessRoleRoutes.ts`, `workspaceConfigRoutes.ts`, `auditRoutes.ts`.

Транспорт: прод-страницы оборачивают surface в `<AdminRuntimeProvider live>` → боевой `createAdminClient` на `/api/*` (cookie-сессия, прокси next.config). `mock-admin-backend` реальными страницами **не** используется (только stories без провайдера) — но см. замечания: баннеры на live-страницах утверждают обратное.

Учётки (см. `docs/qa/full-eval/env-diff.md`):
- **admin** — admin@kiss-pm.local / admin12345 (tenant-alpha, full-профиль)
- **beta-admin** — beta@kiss-pm.local / beta12345 (tenant-beta, тенантная изоляция)
- **plan-reader** — plan-reader-no-resources@kiss-pm.local / reader12345 (не-админ)
- **resource-reader** — resource-reader@kiss-pm.local / resource12345 (не-админ)

## Замечания по коду

1. **mock-admin-backend реальными страницами не используется** — все 4 прод-страницы (`apps/web/src/app/admin/*/page.tsx`) оборачивают surface в `<AdminRuntimeProvider live>`, и `use-admin.ts`/`use-security-policy.ts`/`use-audit-events.ts` при `live` создают клиент без `fetchImpl` (боевой fetch на `/api/*`). Мок активен только в stories (без провайдера). Однако `createMockAdminFetch` импортируется безусловно и попадает в прод-бандл; а surface, отрендеренный где-либо без провайдера, молча упадёт в мок (default `live:false`) — тихий фейковый режим по умолчанию.
2. **Лживые «Прототип»-баннеры на live-страницах** (`users-surface.tsx:48-51`, `roles-surface.tsx:74-77`, `security-surface.tsx:82-85`, `audit-surface.tsx:26-29`): текст утверждает «in-memory mock, swap = apiOrigin», у ролей — «Данные in-memory», хотя страницы работают на боевом API. Нарушение прототип-vs-real маркировки в обратную сторону: пользователь думает, что меняет мок, а мутации реальные.
3. **Фейковые контролы каркаса** (`admin-frame.tsx`, `workspace-shell.tsx`): табы админки и вся левая навигация — некликабельные `<span title="Демо-прототип…">`; поиск `disabled`; индикатор «Сохранено» (зелёная пульсирующая точка) захардкожен и не связан ни с каким состоянием. При этом сами роуты /admin/roles, /admin/security, /admin/audit существуют и живые — из UI до них не доехать.
4. **Нет UI-гарда доступа в web**: ни middleware, ни серверной проверки прав на страницах — не-админ получает полный admin-каркас, а отказ приходит только как 403 от API. 403-reason-коды (`decision.reason`) отсутствуют в RU-карте `adminErr` (`admin-bits.tsx`) → пользователь увидит сырой код. Enforcement на API есть (canManageTenantUsers / canRead-/canManageAccessProfiles / canRead-/canManageWorkspaceConfig / canReadAuditEvents + self-гварды) — проверить обязательно под обеими не-админскими учётками.
5. **Реактивация пользователя невозможна из UI**: кнопка деактивации показывается только активным, у EditUserDialog нет поля статуса — деактивированный навсегда «Неактивен» для оператора, хотя API (`UserUpdateInput.status`) поддерживает возврат в active.
6. **Аудит-лог без фильтров, пагинации и деталей**: хук вызывается с фикс. `limit=50`, API поддерживает только `?limit`; строки не раскрываются, актор события не показывается. Позиции ADM-022…024 — это фиксация gap'ов.
7. **Удаление роли без подтверждения** (`roles-surface.tsx:103`): единственная защита от misclick — title-подсказка; деструктивное действие в один клик.
8. **Риск декоративных security-настроек**: включение 2FA/SSO/whitelist сохраняется и пишется в аудит, но нужно проверить, применяются ли они где-то (вход, создание пользователя, тайм-аут сессии). Если нет — это фейковые контролы уровня продукта (кандидаты ADM-017.2, ADM-019.3).
9. Асимметрия контракта (задокументирована в `admin-client.ts`): создание роли — `POST /api/tenant/current/access-profiles`, остальное — `/api/workspace/access-roles`; деактивация — PATCH `status:"inactive"` (отдельной ручки нет). `DELETE /api/workspace/users/:id` существует на API (с self_user_delete_forbidden), но UI удаления пользователей не имеет.
10. Слаг id роли из кириллического названия (`roles-surface.tsx:36-40`) вырезает всё → fallback `role-<base36 timestamp>`; известный кейс коллапса решён суффиксом, но id получается непрозрачным — проверить двойное быстрое создание.
