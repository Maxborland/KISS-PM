# Инвентарь: Коммуникации

Область: `/communications/channels`, `/communications/chat`, `/communications/meetings`, `/communications/notifications`, `/communications/calls`, `/calls/[roomId]`.

Источники: `apps/web/src/app/communications/**`, `apps/web/src/app/calls/[roomId]/page.tsx`, `apps/web/src/communications/**` (surface-компоненты, `comms-client.ts`, `use-comms.ts`, `use-realtime.ts`, `comms-frame.tsx`), `apps/web/src/views/screens/call-runtime-view.tsx`, `apps/web/src/lib/call/*` (call-client, call-engine, use-lobby-preview).

Транспорт: все прод-страницы оборачивают surface в `<CommsRuntimeProvider live>` → боевой API `/api/workspace/*` (cookie-сессия). `mock-comms-backend` реальными страницами **не используется** (только stories-фолбэк), но включается **по умолчанию при отсутствии провайдера** — см. Замечания.

Контекст медиа: LiveKit в локальном окружении **выключен (opt-in)**. Control-plane звонков (комнаты, сессии, события, join-token, участники) — реальный и должен работать; аудио/видео тестируется до границы «нет медиа-провайдера», критерий приёмки — понятная деградация (сообщение, не белый экран).

Роли для прогона: `admin@kiss-pm.local/admin12345` (manage), `engineer@kiss-pm.local/engineer12345` (участник), читатели — ограниченные профили (read-only / без доступа к сущности → ожидание ветки `forbidden`).

## Итоги прогона (2026-07-04, стенд 127.0.0.1:3000 → :4020)

**29 ✅ / 3 ❌ / 14 🚧.** Последний заведённый: **BUG-COMM-06**.

Ключевая находка: `/communications/chat`, `/communications/meetings`, `/communications/calls` (списочные позиции) блокируются уже известным **BUG-009** — hardcoded scope `project/proj-portal` не резолвится (в этом окружении такого проекта нет; реальные id вида `project-<uuid>`). Подтверждено напрямую: `GET /api/workspace/conversations?entityType=project&entityId=proj-portal` → 404 `collaboration_entity_not_found`; тот же запрос с реальным `project-12f33619-…` → 200. Это фронтенд-баг (surface-компоненты не получают `entityType/entityId` пропсами на прод-роутах — см. Замечание №6), сервер работает корректно. Не дублирую отдельным BUG-COMM, помечаю позиции 🚧 со ссылкой на BUG-009.

Для позиций, заблокированных BUG-009, где было возможно, проверил соответствующий контракт напрямую через `fetch` с реальным `entityId` (создание встречи/канала звонка, notes, action items, external-links, pin/reactions, read-state) — результаты контракта отражены в статусах ниже, но сама **UI-позиция** (кнопка/форма на странице) помечена 🚧, так как через штатную навигацию недостижима.

Кандидат «хардкод ME="u-anna"» (COMM-010/012) подтвердить на рантайме не удалось — вся `/communications/chat` не рендерит сообщения из-за BUG-009 (страница падает в error-card раньше monтирования chat-surface). Остаётся установленным фактом по чтению кода (`chat-surface.tsx:30`), не runtime-подтверждённым в этом прогоне.

## Замечания по коду

1. **Хардкод текущего пользователя в чате** — `apps/web/src/communications/chat/chat-surface.tsx:30`: `const ME = "u-anna"`. На live «свои» сообщения (пункт «Изменить»), подсветка и toggle своих реакций определяются по этому id, а не по сессии. Под любым пользователем, кроме u-anna, UI редактирования своих сообщений не появится, а снятие своей реакции не сработает (повторный клик уйдёт как POST-upsert). Признано в комментарии («нет ручки „кто я"»), но это прямой баг-кандидат для live-прогона. **Рантайм-статус: не подтверждено — заблокировано BUG-009 (см. итоги прогона), см. COMM-010/012.**
2. **Табы CommsFrame — не навигация** — `apps/web/src/communications/ui/comms-frame.tsx:54-68`: табы Чат/Каналы/Звонки/Встречи/Уведомления — `<span>` с подсказкой «переключение разделов появится в приложении». Переход между /communications/* возможен только вводом URL. По принятому стандарту («каждый контрол — к реальному эндпоинту», роуты существуют) — халтура/баг-кандидат. Там же фейковый индикатор «Сохранено» (постоянно зелёный, ни к чему не привязан). **Рантайм-статус: ПОДТВЕРЖДЕНО на всех проверенных страницах (channels/chat/notifications) — неактивные табы без tooltip-класса «Демо-прототип», индикатор «Сохранено» статичен на протяжении десятков мутаций. См. BUG-COMM-02 и BUG-COMM-03.**
3. **Баннеры «Прототип» врут на live** — channels/chat/meetings/calls-surface показывают «Данные in-memory; realtime появится в приложении» безусловно, хотя прод-роуты работают на live API, а realtime (SSE) в чате уже подключён. Копирайт не разведён по режиму `useCommsRuntime()`. **Рантайм: присутствует на /communications/channels — тот же известный BUG-014 (баннер «Прототип» на проде), не дублирую отдельным ID.**
4. **mock-comms-backend**: реальными страницами не используется (все 5 роутов + /settings оборачивают в `<CommsRuntimeProvider live>`). Но дефолт контекста — `live: false` (`comms-runtime.tsx:13`): любая новая прод-страница, забывшая провайдер, молча уедет на in-memory мок без единого предупреждения. Риск-кандидат для регресса; клиенты — модульные синглтоны (`use-comms.ts:116-125`), режим фиксируется на первый монтаж.
5. **Уведомления: перехода нет** — `notifications-surface.tsx:200`: `n.route` выводится текстом, ни `<Link>`, ни onClick. Позиция «переходы из уведомлений» невыполнима до доработки (COMM-031). **Рантайм-статус: ПОДТВЕРЖДЕНО, см. BUG-COMM-04.**
6. **Demo-scope захардкожен** — chat/meetings/calls-surface по умолчанию scope `project/proj-portal` (пропсы entityType/entityId прод-роуты не передают). На live пользователь видит коммуникации только демо-проекта; чужой тенант без proj-portal получит пусто/403. **Рантайм-статус: ПОДТВЕРЖДЕНО — это BUG-009, `proj-portal` не существует как реальный проект в этом окружении (реальные id — `project-<uuid>`), 404 collaboration_entity_not_found на всех трёх surface (chat/meetings/calls).**
7. **Деградация /calls/[roomId] без LiveKit** — корректная по критерию: доступ проверяется до включения камеры (fail closed), ошибка подключения даёт `BannerInline` «Не удалось подключиться к звонку» + `room.disconnect()` (нет «горячего микрофона»), белого экрана нет. Слабые места: (а) 404/403 неразличимы («Нет доступа» и для несуществующего roomId); (б) код ошибки (`video_provider_unavailable`) не показывается — пользователь не узнает, что дело в отключённом медиа-провайдере; (в) лобби всё равно запрашивает камеру ДО попытки подключения — при выключенном LiveKit пользователь пройдёт лобби впустую. **Рантайм-статус: ПОДТВЕРЖДЕНО целиком (создал реальные call-room'ы с provider=manual и provider=livekit через API, прошёл лобби → Присоединиться → корректный баннер ошибки). Join-token вернул 501 Not Implemented (не 409 из описания) — расхождение статус-кода, не критично для критерия деградации.**
8. **resolveActiveSessionId по окну 50 событий** — `calls-surface.tsx:384-390`: активная сессия ищется в последних 50 событиях; у «шумной» комнаты session_started может выпасть из окна → кнопки управления активной сессией исчезнут при status=active. Edge для длинных звонков. Не проверено (нужна комната с 50+ событиями).
9. **Pin без unpin** — контракт не имеет unpin-роута (`comms-client.ts:517`), закрепление из UI необратимо. Кнопки «Удалить»/«Закрепить» показываются всем (RBAC — только на сервере): под ограниченной ролью ожидаем вежливое «Отклонено», это надо явно прогнать. **Рантайм-статус: ПОДТВЕРЖДЕНО — DELETE .../pin → 404. См. BUG-COMM-06.**
10. **markNotificationRead не идемпотентен** (`comms-client.ts:681-684`): повторный POST перезаписывает readAt. «Прочитать все» = N последовательных POST — на большой пачке возможны частичные сбои (UI их считает, но не ретраит). **Рантайм-статус: ПОДТВЕРЖДЕНО. См. BUG-COMM-05.**
11. **Auto-markRead в чате** — ChatPane при входе в беседу с непрочитанными сам вызывает markRead (`chat-surface.tsx:329-334`): кейс «0 непрочитанных» надо готовить с учётом того, что счётчик обнуляется входом. Не проверено (chat-surface недостижим, BUG-009).
12. **XSS**: тела сообщений/заметок/уведомлений рендерятся как React-текст (`{m.body}` + whitespace-pre-wrap) — инъекция HTML не исполняется; `dangerouslySetInnerHTML` в области нет. Прогнать формально (COMM-007/009/024/045). **Рантайм-статус: ПОДТВЕРЖДЕНО безопасно на COMM-007 (канал), COMM-024 (заметка встречи, через API), COMM-045 (чат звонка) — во всех случаях payload не исполнился.**
13. **Записей звонка из UI не добавить** — хук `addRecording` есть (`use-comms.ts:339-342`), кнопки в calls-surface нет (COMM-041); также нет UI приглашения участника (событие participant_invited есть в словаре таймлайна, ручки/кнопки нет). **Рантайм-статус: ПОДТВЕРЖДЕНО (recordings: [] в детали комнаты, кнопки нет в runtime-view).**
14. **Meetings: валидация только finish>start** — создание встречи целиком в прошлом клиент не запрещает; поведение сервера надо зафиксировать (COMM-021). Календарного представления нет (COMM-027). **Рантайм: поле для даты — `scheduledStart/scheduledFinish` (не `start/finish`, как можно было бы предположить из названия критерия) — на это стоит обратить внимание при дальнейшей автоматизации тестов.**
