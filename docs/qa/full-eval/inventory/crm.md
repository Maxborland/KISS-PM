# Инвентарь: CRM

Область: `/crm/clients`, `/crm/contacts`, `/crm/deals`, `/crm/deals/[id]`, `/crm/products` (Next.js app router, `apps/web/src/app/crm/**`).
Поверхности: `apps/web/src/crm/{clients,contacts,deals,products}/*-surface.tsx`, клиент API `apps/web/src/crm/lib/crm-client.ts`, хук `use-crm.ts`, транспорт `crm-runtime.tsx`.

Все 5 прод-страниц оборачивают surface в `<CrmRuntimeProvider live>` → реальный API `/api/workspace/*` (cookie-сессия, прокси next.config). Mock-бэкенд (`mock-crm-backend.ts`) активен только без провайдера (Storybook) — прод его **не** использует. Но прототип-маркеры («Данные in-memory», некликабельные табы) остались в живых страницах — см. «Замечания по коду».

Роли для проверки:
- **admin** — `admin@kiss-pm.local` / `admin12345` — всё доступно.
- **engineer** — `engineer@kiss-pm.local` — полный профиль (ожидается доступ к CRM).
- **читатели плана/ресурсов** — НЕ должны иметь CRM-прав: каждый роут обязан показать forbidden-экран (API отдаёт 403 из `canRead*`/`canManage*`, хук маппит в `status="forbidden"`).

## Замечания по коду

1. **Mock-бэкенд прод не использует** — все 5 страниц (`apps/web/src/app/crm/*/page.tsx`) рендерят surface в `<CrmRuntimeProvider live>`; `createMockCrmFetch` подключается в `useCrmClient` только без провайдера (Storybook). Однако **прототип-маркеры остались в live**: на каждой поверхности безусловный баннер «Прототип … Данные in-memory» — на боевом роуте это ложь (данные в Postgres). Баг-кандидат по стандарту «строгий Storybook + полный API».
2. **Табы CrmFrame — декорация** (`apps/web/src/crm/ui/crm-frame.tsx`): «Сделки/Клиенты/Контакты/Продукты» — `<span>` с тултипом «Демо-прототип: переключение разделов появится в приложении». На прод-роутах между /crm/* нельзя перейти из UI. Плюс постоянный фейковый индикатор «Сохранено» (зелёный пульс) — имитация контроля.
3. **Несуществующий deal id не даёт 404** (`deal-card-surface.tsx:73-76`): `/crm/deals/мусор` молча показывает `opp-2207` или первую сделку. Карточка вообще не делает `GET /opportunities/:id` — грузит весь список и ищет в нём.
4. **Нет поиска, фильтров и сортировки ни на одной CRM-таблице** — позиции из чек-листа «фильтры/поиск/сортировка» физически отсутствуют в UI (клиенты, контакты, продукты, список сделок). Gap для инвентаря, а не тест.
5. **Редактирование = только архивация**: у клиентов/контактов/продуктов нет формы правки полей (name/email/цена), хотя PATCH full-replace на API есть. «Редактирование клиента-контакта-продукта» протестировать нечем, кроме архива.
6. **Смена стадии из карточки сделки идёт через PATCH /opportunities/:id (full-replace), а не через /:id/stage** — потенциальный обход гвардов переходов воронки (transition_not_allowed/condition_*). Требует проверки на API: применяет ли `updateOpportunityCommand` те же правила, что `changeOpportunityStageCommand`.
7. **CreateDealDialog: захардкоженные значения** (`deals-surface.tsx:404-416`): demand — фиксированная позиция `positionId:"backend"` (на боевом тенанте может не существовать → feasibility даст missing_position_capacity); дефолтные даты `2026-05-01/2026-08-01` (протухнут); `projectTypeId` берётся молча как первый из списка — селектора нет.
8. **Дубликат email контакта не блокируется**: `crmParsers.ts` только lowercase + формат-regex, уникальности нет — два контакта с одним email создаются.
9. **`href={a.fileUrl}` без валидации схемы** в ленте активности (`deal-card-surface.tsx:374`) — file-активность с `javascript:`-URL — XSS-вектор, если сервер не фильтрует схему при POST .../files.
10. **Расхождение money()**: в `deals-surface.tsx:22` локальный `money` без ветки `v<1000` (сумма 500 ₽ → «0 тыс ₽» на канбане), тогда как общий `crm-bits.money` эту ветку имеет.
11. **Частичная деградация прав**: 403 на `/api/workspace/projects` глотается (пустой список — осознанно), но 403 на `stage-transitions` любой воронки роняет весь раздел «Сделки» в forbidden/error (`use-crm.ts:79-82`); ошибки `/users` глотаются молча (имена → сырые id). Плюс N+1: stage-transitions грузятся отдельным запросом на каждую воронку.
12. **Финализация без причины от пользователя**: карточка сделки шлёт захардкоженные reason («Закрыта вручную»/«Отказ клиента»); в воронке `deals-surface` finalize вообще не вызывается из UI.
13. **Пайплайн-унификация**: web продолжает ходить в legacy-проекцию `GET /api/workspace/deal-stages` (после унификации crm_pipeline_* она осталась на API — `openApiDocument.ts:79-81`); стадии с `pipelineId:null` (legacy) не попадают ни в одну воронку канбана и видны только колонкой «Без стадии» через сделки. Проверить на живых данных, что после миграции нет стадий-сирот.
14. Копейки невозможны by design: `contractValue`/`price` — положительные целые ₽ на сервере, UI делает `Math.round` перед отправкой — суммы с копейками молча округляются без предупреждения.
