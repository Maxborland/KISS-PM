# Инвентарь: Аутентификация

Область: `/login`, `/register`, `/password-reset`, `/password-reset/confirm`, logout, защита роутов, сессионные куки.
Источники: `apps/web/src/app/(auth)/**`, `apps/web/src/auth/**` (login-surface.tsx, register-surface.tsx, reset-request-surface.tsx, reset-confirm-surface.tsx, auth-client.ts, use-auth.ts, auth-runtime.tsx, auth-bits.tsx, profile-surface.tsx — для logout). Прод-роуты монтируют surface в `<AuthRuntimeProvider live>` → боевой API через rewrites `next.config.ts` (`/api/:path*` → `KISS_PM_API_ORIGIN`), cookie-сессия HttpOnly.

Тестовые учётки: `admin@kiss-pm.local/admin12345` (tenant-alpha), `beta@kiss-pm.local/beta12345` (tenant-beta), `engineer@kiss-pm.local/engineer12345`, `plan-reader-no-resources@kiss-pm.local/reader12345`, `resource-reader@kiss-pm.local/resource12345`.
⚠️ В коде поверхностей зашита ДРУГАЯ демо-пара `admin@kiss-pm.local / kiss-pm-admin` (пароль мока, не боевого сида) — при проверках на live использовать учётки выше.

> Прогон 2026-07-04, два захода. Первый заход шёл через чужой worktree-стенд (API v3-backend-unblock, trusted origin :3001) — все браузерные мутации кроме login падали 403 `same_origin_action_required` (зафиксировано как BUG-AUTH-01, после пересборки стенда переквалифицирован в ⚪ артефакт окружения). Второй заход — чистый стенд web :3000 → API :4020 (текущий чекаут, дефолтные trusted origins): logout/register/password-reset из браузера работают; статусы ниже отражают чистый стенд.

## Замечания по коду

Транспорт: прод-роуты честные — все четыре страницы `(auth)/*` монтируют surface в `<AuthRuntimeProvider live>` → реальный `fetch` на `/api/auth/*` через rewrite `next.config.ts` (mock-auth-backend в проде НЕ используется, он только для Storybook/тестов). «Фейкового контрола» уровня mock-вместо-API нет, но есть пласт Storybook-артефактов, протёкших в прод:

1. **Нет защиты роутов вообще** — в `apps/web/src` нет `middleware.ts`, нет layout-guard, нет ни одного redirect на /login. Защищённые страницы (/dashboard, /projects, /crm, /admin…) монтируются анонимно, данные просто падают 401. Только /profile показывает осмысленный forbidden. Кандидат: критический.
2. **Нет редиректа после логина** — `login-surface.tsx:139-141` («Честная заглушка редиректа: в Storybook навигации нет») рендерится и на прод-роуте /login: после успешного входа пользователь остаётся на /login с текстом «В приложении здесь — переход в рабочую область». Кандидат: критический.
3. **Storybook-ссылки на прод-роутах** — `reset-request-surface.tsx:67` и `:144`, `reset-confirm-surface.tsx:63` и `:84`: `href="?path=/story/…"` (навигация Storybook) уходит в прод как мусорный query-параметр. Сквозной флоу сброса пароля кликами не проходится.
4. **Мёртвые контролы навигации**: /login футер «Создать аккаунт»/«Забыли пароль?» — инертные `<span>` (login-surface.tsx:158-169), хотя /register и /password-reset существуют; /register «Войти» → `href="#login"` (register-surface.tsx:86) — мёртвый якорь.
5. **Плашки «Прототип» с ложной копией на проде** — все четыре surface рендерят плашки «Contract-mock… данные in-memory», хотя транспорт live; на /login плашка вдобавок публикует демо-креды `admin@kiss-pm.local / kiss-pm-admin` (login-surface.tsx:174-186).
6. **Предзаполненный админский email на /password-reset** — `reset-request-surface.tsx:32`: `useState("admin@kiss-pm.local")` — прод-страница сброса пароля предзаполнена админским адресом.
7. **devToken в контракте** — `use-auth.ts:168-172` кастит ответ request к `{devToken?: string}`; боевой API его не отдаёт, но если когда-нибудь отдаст (флаг dev на сервере) — UI покажет токен сброса кому угодно (DevTokenPanel). Проверить на live, что 202-ответ чист.
8. **/register при живой сессии показывает «Аккаунт создан»** — `register-surface.tsx:41`: `registered = state==="authenticated"` истинно и для ранее вошедшего пользователя, регистрации не совершавшего.
9. **Несимметричная нормализация email** — login лоуэркейсит (`login-surface.tsx:60`), register/reset — нет: возможен кейс `Admin@…` при регистрации → потом невозможность входа (login шлёт lowercase). Зависит от нормализации на сервере — проверить.
10. **Клиентская валидация неравномерна**: /login проверяет формат email regex'ом, /register и /password-reset — нет (только непустоту / вообще ничего). Не баг сам по себе (сервер валидирует), но UX-расхождение.
11. **«Войти снова» после logout на /profile** — `window.location.reload()` вместо перехода на /login (profile-surface.tsx:107) → перезагрузка возвращает тот же forbidden.
12. **Logout отсутствует в WorkspaceShell** — грепом по `apps/web/src/delivery` кнопки «Выйти» нет; выйти можно только с /profile (или /login). Функциональное меню аватара с logout (`avatar-menu-surface.tsx`) существует, но в прод-роуты НЕ смонтировано (только Storybook, с авто-логином демо-кредами внутри — в прод такое монтировать нельзя без переделки).
13. **Расхождение демо-кред и сида**: в коде `kiss-pm-admin`, тестовые учётки стенда — `admin12345`. Источник путаницы при ручном тестировании.
14. **UI-состояние ошибок хука `useAuth`**: mutation-ошибки маппятся `message: e.code` (use-auth.ts:60) — при неизвестном коде FormError покажет сырой код (authErr fallback на code). Проверить на нестандартных ответах (HTML-страница 502 от прокси → `invalid_json_response` → «Ошибка»?  код не в ERR-словаре).
