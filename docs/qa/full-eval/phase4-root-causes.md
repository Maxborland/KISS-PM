# Фаза 4 — анализ первопричин и план связных фиксов

Loop #010, шаг 4: «Разобрать весь набор багов на общие причины, зависимости и конфликтующие фиксы; реализовать наименьшее связное решение с регрессионным покрытием.»

Всего зафиксировано **70 багов** (auth 13, shell-dashboard 15, projects 22, crm 9, communications 6, admin 5); из них 3 auth-бага переквалифицированы в ⚪ артефакты окружения. Ниже — не 70 независимых дефектов, а **12 первопричинных кластеров**. Чинить надо причины, не симптомы.

---

## Кластер A — Прототип-обёртка не отключена для прод-режима
**Баги:** BUG-014, BUG-005, AUTH-08, SHELL-15, PROJ-12, +баннеры на CRM/Comms/Admin, фейковый индикатор «Сохранено» (SHELL, COMM-03, ADM), демо-креды на /login (AUTH-08).
**Причина:** surface'ы несут прототип-хром (баннеры «Прототип/contract-mock/данные in-memory», «v0.1 · прототип», статичный «Сохранено», демо-креды), который рендерится и в live-режиме. Данные при этом реальные из Postgres — маркеры лгут.
**Фикс:** один флаг/условие рендера прототип-блоков, завязанное на режим (live vs story). Скрыть в прод-сборке. Небольшой связный фикс.
**Замечание пользователя:** прямо просил убрать «блоки прототип/демо из прод».

## Кластер B — Оболочка/табы — демо-хром, навигация не подключена
**Баги:** BUG-002, SHELL-02, PROJ-06/07/08, COMM-02, ADM-01(часть), CRM(табы), command palette (SHELL-03).
**Причина:** WorkspaceShell / *Frame — сторибучные версии: пункты сайдбара и табы — `<span>` без href/onClick; фейковые бейджи (12/8/37/42); поиск disabled; аватар хардкод «КБ»; Ctrl+K не смонтирован. До существующих роутов из UI не добраться.
**Фикс:** заменить демо-хром на реальную навигацию (Link/router), убрать фейковые бейджи или завести реальные. **КРУПНЫЙ** — продуктовое решение, не мех-правка. → кандидат в blocked-handoff.

## Кластер C — Мок-данные протекают в live-поверхности
**Баги:** SHELL-04 (CRM-плитки дашборда из mock-crm — нет CrmRuntimeProvider), PROJ-02/17/20 (селекты исполнителей из статического RESOURCES → 409), SHELL-12 (agent propose из mock-llm), PROJ-05/04 (мок-эпоха дат, BASE_MS).
**Причина:** compоненты берут демо-данные (`planning-demo-data.ts` RESOURCES, mock-crm-backend, mock-llm) вместо live-справочников.
**Фикс:** заменить импорты мок-справочников на live-хуки (`useResourceDirectory`/`GET /api/workspace/users`, CrmRuntimeProvider на дашборде). Средний, но связный по кластеру.

## Кластер D — Хардкод невалидных идентификаторов в командах/пейлоадах
**Баги:** PROJ-01 (`statusId:"todo"` → создание задачи 409), CRM-05 (`positionId:"backend"` → создание сделки 500), BUG-009 (`entityId=proj-portal` → 14 comms-позиций 404), ME="u-anna" в chat-surface.
**Причина:** UI шлёт зашитые демо-id вместо резолва из тенанта/контекста.
**Фикс:** резолвить статус/позицию/scope/currentUser из реальных источников. Ломает основные create-флоу — **высокий приоритет**. Каждый — точечный, но требует подтягивания справочника.

## Кластер E — Нет защиты роутов и редиректа после логина
**Баги:** AUTH-02 (аноним монтирует /dashboard, /admin — нет middleware), AUTH-03 (нет редиректа после входа), SHELL-03 (admin виден всем ролям).
**Причина:** в apps/web нет middleware/guard; после login surface не редиректит.
**Фикс:** Next middleware, проверяющий сессию + редирект. Средний, безопасностно-значимый.

## Кластер F — Сырые коды ошибок вместо RU-текста
**Баги:** SHELL-06, ADM-01, PROJ-15, CRM(нотисы), AUTH-12(латентно), ADM-05 (устаревшие ключи audit-меток), AUTH-13 (reset_token_used не отдаётся).
**Причина:** карты код→RU неполны/устарели; неизвестный код утекает сырым.
**Фикс:** дополнить словари + фолбэк на общий RU-текст для неизвестного кода. Мелкий, безопасный, хорошо тестируемый.

## Кластер G — Сырые технические id вместо имён
**Баги:** BUG-013/007 (position-engineer, project.demand, «дэмэнд»), PROJ-14 (user-id под PR), CRM-07, SHELL-07 (projectId/taskId в my-work).
**Причина:** UI рендерит id/типы вместо резолва через справочники позиций/пользователей/проектов + локализацию.
**Фикс:** резолв через справочники (часть пересекается с кластером C).

## Кластер H — Молчаливые no-op мутации / потеря данных (backend/repo)
**Баги:** PROJ-03 (`task.update_progress` — 200+audit+version, но значение теряется), PROJ-18 (фантомное снятие исполнителя), PROJ-19 (`accept_overload` — `planningRepository.ts:1117 return;` пустышка), PROJ-22 (baseline ложные Δ), PROJ-24 (revert недостижим на /commits — `lastApplyRef` per-component, всегда пустой), PROJ-25 (scenario reassignment берёт «первый другой ресурс» без фильтра capacity/role).
**Причина:** команды применяются формально (версия растёт, аудит пишется), но эффект не сохраняется — самый опасный класс (целостность данных).
**Фикс:** доимплементировать редьюсеры/репозиторий. Требует аккуратности + регрессы. **Высокий приоритет по риску.**

## Кластер M — Preview валидирует ВЕСЬ план, а не команду + битая seed-веха (BUG-PROJ-23, CRITICAL)
**Баг:** PROJ-23 — одна предсуществующая ошибка валидации плана (seed-веха с `invalid_work_model` на project-vektor-portal и project-gorset-migration) НАВСЕГДА блокирует ЛЮБУЮ команду планирования на этих проектах (schedule/scenarios/baseline/assignments/calendars/settings). project-demo-crm-intake (0 предошибок) не затронут — доказывает механизм.
**Две первопричины:**
1. **Данные:** seed создаёт веху с `durationMinutes<=0`, а `resolveDurationMinutes` (`schedulingEngine.ts:254`) флагует любую задачу с `durationMinutes<=0` как `invalid_work_model` — вехи (нулевая длительность) не исключены из проверки.
2. **Логика:** `previewPlanningCommand`/`previewPlanningCommands` (`planningRouteHelpers.ts`) блокируют команду, если в ПОЛНОМ пересчитанном плане есть ЛЮБАЯ error-issue, а не только связанная с этой командой.
**Влияние:** это КОРЕНЬ значительной части ранних 409 в планировщике на vektor-portal (часть наблюдений projects-A). Объясняет, почему «всё падает» на seed-проектах, но работает на чистом demo-crm-intake.
**Фикс:** (a) быстрый — починить seed-веху (durationMinutes=null для вех) → разблокирует среду немедленно; (b) корневой — либо исключить вехи из invalid_work_model, либо скоупить preview-валидацию на команду. Оба трогают ядро домена → корневой в blocked-handoff, быстрый data-фикс возможен.

## Кластер N — 403 глотается без обработки (unhandled rejection → «пусто»)
**Баги:** PROJ-26 (`/commits` PR: 403 на audit-events → «История пуста», неотличимо от реально пустого), SHELL-05/08, CRM(users glotается). Класс BUG-PROJ-15.
**Фикс:** `.catch` на запросах справочников/аудита + различать 403 от пустого результата. Мелкий, но размазан по многим surface.

## Кластер I — Деструктив без подтверждения / гардов
**Баги:** ADM-04 (удаление роли в один клик), ADM-02 (обнуление прав назначенной роли молча — реально лишило resource-reader всех прав).
**Фикс:** confirm-диалог + предупреждение при снятии прав у назначенной роли. Мелкий, важный.

## Кластер J — 500 вместо 4xx на нарушении ограничений
**Баги:** CRM-01 (дубликат SKU → 500), CRM-05 (создание сделки → 500).
**Фикс:** ловить конфликт БД → 409/422 с RU-текстом. Мелкий, тестируемый.

## Кластер K — Декоративные security-настройки без enforcement
**Баги:** ADM-03 (2FA/SSO/whitelist сохраняются и аудируются, но не применяются).
**Фикс:** enforcement на login/create-user. **Крупный**, безопасностный. → кандидат в blocked-handoff (или явно пометить настройки как «планируется»).

## Кластер L — Недостающие фичи/эндпоинты
**Баги:** COMM-06 (нет unpin), COMM-04 (нет перехода из уведомления), SHELL-09 (тема не применяется), SHELL-10 (нельзя очистить телефон — UI шлёт null, API ждёт ""), ADM (реактивация пользователя), scenarios (частично).
**Фикс:** точечные фичи; вне «наименьшего связного решения» — по одному.

## Зависимости и конфликты фиксов
- **B (навигация) блокирует наблюдаемость почти всего** — но её фикс крупный; тестировали через прямые URL, так что остальные фиксы не ждут B.
- **C и G пересекаются** (общий live-справочник) — чинить одним заходом.
- **D и H оба про планировщик** — не конфликтуют, но трогают близкий код (planningCommands/reducer/repository); делать одной серией с общими регрессами.
- **A (прототип-флаг)** независим и безопасен — хороший первый фикс.
- **F, J, I** — мелкие изолированные, низкий риск.

## Приоритеты фазы 4 (наименьшее связное, максимум эффекта)
1. **A** — снять прототип-блоки в прод (прямой запрос пользователя, безопасно).
2. **F + J** — RU-словари ошибок + 500→409/422 (мелко, тестируемо).
3. **D** — расхардкодить статус/позицию/scope в create-флоу (чинит мёртвые основные сценарии).
4. **H** — no-op мутации планировщика (риск целостности).
5. **I** — гарды деструктива в админке.
6. **Blocked-handoff:** B (переписать навигацию оболочки), K (enforcement security), крупные части C/L — продуктовые решения, зафиксированы для передачи.

Каждый применённый фикс — с регрессионным тестом; после серии — фаза 5 (полный повторный прогон).
