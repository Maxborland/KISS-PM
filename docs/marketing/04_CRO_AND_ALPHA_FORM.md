# CRO и форма закрытой альфы

## Тип страницы

Лендинг закрытой альфы для проверки agent-first positioning и спроса на project diff review workflow.

## Главная цель конверсии

Посетитель оставляет заявку на доступ.

Главный призыв:

> Запросить доступ

## Допущения по трафику

До фиксации каналов страница должна работать для холодных и теплых посетителей из:

- founder-led outreach;
- LinkedIn и профессиональных постов;
- рекомендаций от PMO/operations консультантов;
- закрытых сообществ project managers и delivery leaders;
- поисковых запросов вокруг AI project management, project planning, resource planning и "как перепланировать проект".

Первый экран объясняет не "еще один PM tool", а новый способ работы: агент готовит изменения, пользователь ревьюит и применяет.

## Тест пяти секунд

Посетитель должен понять:

1. KISS PM - система управления проектами через агента.
2. Агент готовит Сверку, а не применяет изменения без человека.
3. Можно принять все, выбрать hunks, отредактировать или отклонить.
4. Каждое примененное изменение попадает в audit trail.
5. Следующий шаг - заявка на закрытую альфу.

## Иерархия призывов

| Место | Призыв | Задача |
|---|---|---|
| Первый экран | Запросить доступ | Дать путь готовому посетителю |
| Вторичный призыв первого экрана | Посмотреть Сверку | Подвести к главному proof-блоку |
| После Сверки | Запросить доступ | Конвертировать после демонстрации доверия |
| Alpha section | Запросить доступ | Основная отправка формы |
| Подвал | Запросить доступ | Финальный короткий путь |

Один основной призыв ведет к заявке. Вторичный призыв только скроллит к Сверке.

## Форма закрытой альфы

### Рекомендуемые поля

| Поле | Обязательное | Зачем |
|---|---:|---|
| Имя | Да | Персональный follow-up |
| Email | Да | Связь |
| Компания | Да | Контекст |
| Роль | Да | Persona signal |
| Сколько проектов идет параллельно? | Да | Проверка попадания в целевую аудиторию |
| Где сейчас больше всего ручной проектной рутины? | Нет | Качественный сигнал |

### Варианты для поля проектов

- До 5
- 5-15
- 15-30
- 30-50
- 50+

Старый порог 30-150+ не удаляется из стратегического сегмента, но в форме лучше не отсеивать более ранние команды: agent-first боль может проявиться уже на 5-15 клиентских проектах, если много встреч, изменений и ручных follow-ups.

### Варианты роли

- Собственник / директор
- COO / операционный директор
- Head of delivery
- PMO / руководитель проектного офиса
- Project manager
- Resource manager
- Tenant admin / IT
- Другое

### Подсказка к форме

> Нам важно понять, где проектные решения сейчас застревают между встречей, задачами, сроками и статусами.

### Состояние успеха

> Спасибо. Мы получили заявку и свяжемся с вами, когда будем готовы показать закрытую альфу.

### Состояние ошибки

> Не удалось отправить заявку. Проверьте соединение и попробуйте еще раз.

### Текст о приватности

> Используем данные только для связи по закрытой альфе KISS PM.

## Быстрые CRO-улучшения

1. В hero показать разговор с агентом рядом со Сверкой.
2. Не ставить дашборд как главный визуал.
3. В первом экране явно сказать: "AI предлагает, вы применяете".
4. После первого proof-блока показать реальные hunks: задача, срок, владелец, dependency, риск, сообщение.
5. Призыв `Посмотреть Сверку` должен вести к демонстрации, а не к общему описанию возможностей.
6. Форма спрашивает не только масштаб, но и источник ручной рутины.
7. Trust строится через review, partial apply, manual edit и audit.
8. Не обещать autonomous PM, ROI или production maturity.

## Вопросы посетителя и ответы

| Вопрос | Где закрывать | Угол ответа |
|---|---|---|
| "AI сам будет менять проект?" | Первый экран, FAQ, блок Сверки | Нет. Сначала Сверка, потом ревью человеком. |
| "Что именно я ревьюю?" | Блок Сверки | Отдельные изменения: задачи, сроки, владельцы, зависимости, риски, сообщения. |
| "Можно принять только часть?" | Ручной контроль | Да. Главный trust-механизм - partial apply. |
| "Как это работает с Jira/Asana/Bitrix24?" | FAQ | KISS PM продает agent workflow; текущие системы могут быть источниками или интеграциями. |
| "Где audit trail?" | Audit block | Каждое примененное изменение получает before/after, actor и permission result. |
| "Насколько продукт готов?" | Блок закрытой альфы | Закрытая альфа, проверяем workflow и собираем design partners. |

## Тесты лендинга

### Заголовок первого экрана

Гипотеза: headline с агентом и одобрением изменений даст более качественные заявки, чем портфельный headline.

- A: `Управляйте проектами через агента. Применяйте только одобренное.`
- B: `Проектный агент, который готовит следующий шаг.`
- C: `От цели к изменению проекта - через ревью.`

### Визуал первого экрана

Гипотеза: разделенный экран "чат + Сверка" лучше объясняет продукт, чем визуал с дашбордом.

- A: разговор + Сверка + audit preview.
- B: agent conversation full-width with drawer Сверки.
- C: дашборд или control surface с agent overlay.

### Подача демо

Гипотеза: конкретная ситуация "задержка -> proposed diff" понятнее, чем общий "AI PM assistant".

- A: `Перепланируй проект после задержки дизайна`.
- B: `Преврати meeting summary в project update`.
- C: `Найди риски перед клиентским звонком`.

### Трение формы

Гипотеза: вопрос про ручную рутину даст больше ценной discovery-информации, чем вопрос только про количество проектов.

- A: короткая форма с контактами и ролью.
- B: recommended form с количеством проектов и ручной рутиной.

### Текст призыва

Гипотеза: прямой призыв к доступу в альфу лучше "получить демо" на стадии закрытой альфы.

- A: `Запросить доступ`
- B: `Попасть в закрытую альфу`
- C: `Посмотреть agent workflow`

## Измерение

Отслеживать:

- клики основного призыва первого экрана;
- клики `Посмотреть Сверку`;
- scroll depth до блока Сверки;
- form start rate;
- form completion rate;
- распределение по количеству параллельных проектов;
- распределение по ролям;
- повторяющиеся формулировки в поле ручной рутины;
- источники трафика;
- долю заявок, где человек явно описывает meeting-to-task, replanning, status или audit pain.

Для закрытой альфы качество заявок важнее raw submissions.
